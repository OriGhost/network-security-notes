# 网络信息安全课程学习笔记详解

> 本笔记依据 `outputs/extracted_course_materials.md` 与 `outputs/extracted_course_materials.json` 中抽取出的 PPT 与实验文档内容整理。它不是期末冲刺版，而是面向课程进行中的学习材料：按 PPT 标题的自然章节顺序展开，兼顾核心主题、知识点解释、课堂案例、交互步骤和常见误区。

## 使用方法

学习这门课时，不要只记结论。网络信息安全的知识通常由三层组成：第一层是安全目标，如保密性、完整性、认证、可用性、抗抵赖；第二层是实现机制，如加密、哈希、MAC、数字签名、证书、认证协议、防火墙；第三层是具体场景，如 HTTPS、Kerberos、ARP 欺骗、DNS 欺骗、SYN Flood、区块链交易。

阅读本笔记时建议按如下方式理解：

1. 先看每章“本章主线”，知道这一章解决什么问题。
2. 再看“关键知识”，把概念、公式、协议流程和案例对应起来。
3. 对涉及 A/B/C、客户端/服务器、CA/KDC/TGS 等多方交互的内容，重点看步骤顺序和每一步使用的密钥。
4. 最后看“易错点”，这些地方通常也是课堂提问或考试题容易出现的位置。

## 1. 第一章-网络安全引言.pptx

### 本章主线

第一章的核心是建立网络安全的总框架：为什么需要网络安全，安全到底保护什么，攻击有哪些类型，安全机制如何对应安全服务。PPT 通过勒索病毒、手机恶意软件、网站篡改、ARP 欺骗、拒绝服务、漏洞攻击等例子说明：网络安全不是某一个软件或某一道防线，而是协议、系统、管理、人员和法律共同构成的系统工程。

课程目标可以概括为五点：掌握网络安全基本概念，理解基础密码算法及应用，掌握网络攻防基本原理，了解前沿技术，并能动手完成简单实验。后续 PPT 中的密码学、PKI、身份认证、Web/TLS、TCP/IP 安全、攻击、防火墙和区块链，都可以看作第一章框架下的展开。

### 网络安全的基本目标

保密性是指信息不被未授权主体看到。例如网银口令、转账金额、通信内容不能被旁路监听者获得。实现手段包括对称加密、公钥加密、访问控制和网络隔离。

完整性是指信息不被未授权修改、插入、删除、重放或乱序。例如转账 1000 元不能被改成 10000 元。常见机制包括哈希、MAC、数字签名、校验、审计日志。

可用性是指合法用户在需要时能够使用服务。例如 Web 服务器不能被 DDoS 攻击打到无法响应。可用性防护依赖冗余、限速、流量清洗、隔离、备份和应急响应。

可认证性是指能够确认通信实体或数据来源。例如浏览器需要确认访问的确实是银行服务器，而不是攻击者搭建的钓鱼站。常见机制包括口令、证书、质询-应答、Kerberos。

抗抵赖是指行为发生后不能否认。例如 A 签署合同后不能否认签过。数字签名、时间戳、公证机制和可信第三方是抗抵赖的主要基础。

### 安全威胁与攻击分类

被动攻击只观察或复制信息，不改变系统状态。典型例子是嗅探和通信流量分析。被动攻击难以检测，因为攻击者不一定留下明显痕迹，所以重点是通过加密等方式阻止攻击成功。

主动攻击会改变信息、状态或服务能力。典型例子包括伪装、重放、篡改、拒绝服务、ARP 欺骗、DNS 欺骗。主动攻击通常更容易造成直接后果，也更容易被日志、IDS 或业务异常发现。

PPT 中多次出现的 ARP 欺骗是理解主动攻击的好例子。ARP 协议本身缺少认证，攻击者可以伪造“网关 IP 对应我的 MAC 地址”的响应，使目标主机把流量发给攻击者。这一案例说明：协议设计缺陷会直接导致安全问题。

### OSI 安全体系

OSI 安全体系把安全问题拆成三类：安全攻击、安全服务、安全机制。

安全服务回答“要达到什么安全目标”，包括鉴别、访问控制、数据保密、数据完整性和不可否认。

安全机制回答“用什么技术实现目标”，包括加密机制、数字签名、访问控制、数据完整性、鉴别交换、业务流填充、路由控制和公证机制。

安全攻击回答“攻击者如何破坏目标”，包括被动攻击和主动攻击。学习时要把这三者连起来：例如 TLS 使用证书和握手完成服务器认证，使用对称加密实现保密性，使用 MAC 或 AEAD 实现完整性。

### 网络安全模型

PDR/P2DR 模型把安全看作动态闭环：Policy 制定策略，Protection 进行防护，Detection 发现入侵，Response 响应处理。WPDRRC 又加入预警、恢复和反击等思想。

IATF 强调深度防御，即不能只依赖一层防线。终端、网络边界、传输链路、应用系统、认证授权、审计和恢复都要协同工作。一个典型例子是 HTTPS：它能保护网络传输，但如果客户端已经被木马控制，TLS 之前输入的口令仍可能被窃取。

### 课堂案例

勒索病毒案例体现保密性和可用性的双重破坏：攻击者用加密手段锁住用户文件，再要求赎金。这里加密技术本身不是恶意的，恶意的是攻击者未经授权使用加密破坏可用性。

ARP 欺骗案例体现协议缺少认证的风险。正常 ARP 过程是：主机 A 广播“谁是 IP_B”，主机 B 单播回应“IP_B 的 MAC 是 MAC_B”。攻击时，攻击者伪造响应，让目标把网关 IP 绑定到攻击者 MAC，从而形成中间人。

APT、震网、乌克兰断电等案例说明高级攻击往往不是单点技术，而是钓鱼邮件、凭证窃取、横向移动、工控系统操作、日志破坏、拒绝服务等多步骤组合。

### 易错点

不要把“安全”等同于“装了防火墙”或“用了加密”。防火墙主要控制边界流量，加密主要保护数据内容，它们不能替代身份认证、补丁管理、日志审计、备份恢复和人员管理。

不要说被动攻击一定没有危害。被动攻击虽然不篡改数据，但可破坏保密性，并为后续主动攻击收集情报。

不要把安全机制和安全服务混淆。加密是机制，保密性是服务；数字签名是机制，认证、完整性、抗抵赖是它提供的服务。

## 2. 第二章-传统密码学.pptx

### 本章主线

传统密码学从最基础的密码系统模型讲起，通过 Caesar、单字母表替代、Playfair、Vigenere、Vernam、置换密码和转子机说明两个核心思想：第一，密钥必须承担安全性；第二，密码算法要同时对抗穷举和统计分析。

### 密码系统模型

密码系统包含明文 `P`、密文 `C`、密钥 `K`、加密算法 `E` 和解密算法 `D`。基本形式是：

```text
C = E_K(P)    # P 是明文，K 是密钥，E 是加密算法；发送方用 K 把 P 加密成密文 C
P = D_K(C)    # C 是密文，D 是解密算法；接收方用正确密钥 K 把 C 解回明文 P
```

这两行表示：发送方用密钥 `K` 把明文 `P` 变成密文 `C`；接收方再用正确密钥把 `C` 还原为 `P`。攻击者即使知道算法，如果不知道密钥，也不应能恢复明文。

Kerckhoffs 原则要求：密码系统即使算法公开，也应保持安全；秘密只应依赖密钥。不要依赖“算法没人知道”，因为算法可能被逆向、泄露或标准化公开。

### Caesar 密码

Caesar 密码是单表移位替代：

```text
C = (p + k) mod 26    # p 是明文字母编号，k 是移位密钥；加密时向后移动 k 位，mod 26 表示超过 Z 后从 A 循环
p = (C - k) mod 26    # C 是密文字母编号；解密时向前移动 k 位，恢复原明文字母编号 p
```

这里 `p` 是明文字母编号，`k` 是移位量，`mod 26` 表示超过 Z 后从 A 重新开始。若 `k=3`，则 `A->D`，`B->E`，明文 `HELLO` 会变成 `KHOOR`。它的密钥只有 26 种，攻击者可以直接穷举。

### 单字母表替代

单字母表替代用一张固定替代表把每个明文字母替换成密文字母。密钥空间是 `26!`，看起来很大，但它保留了自然语言的词频。比如英语中 `E` 常出现，如果某个密文字母也高频出现，就可能对应 `E`。

例：若替代表中 `A->Q, B->W, D->R`，则明文 `BAD` 变成 `WQR`。这里每次出现 `B` 都会变成 `W`，所以统计规律不变。

### Playfair 密码

Playfair 一次处理两个字母。它使用 5x5 密钥矩阵，通常合并 I/J。若明文两个字母在同一行、同一列或矩形对角，会按对应规则替换。

例：用 `MONARCHY` 构造矩阵时，`HI` 这组字母可按矩形规则替换为另一对字母。重点不是记某个固定输出，而是理解 Playfair 把单字母频率分析变成双字母组分析，安全性比单字母替代更强，但仍可被分析。

### Vigenere 密码

Vigenere 使用多个 Caesar 字母表。若把字母转成 `A=0` 到 `Z=25`，可写成：

```text
C_i = (P_i + K_i) mod 26    # P_i 是第 i 个明文字母编号，K_i 是第 i 个密钥字母偏移；逐位移位得到第 i 个密文字母 C_i
P_i = (C_i - K_i) mod 26    # C_i 是第 i 个密文字母编号；用同一位置的 K_i 反向移位，恢复 P_i
```

这里 `K_i` 是密钥串第 `i` 位对应的偏移。例：明文 `ATTACK`，密钥 `KEY` 循环为 `KEYKEY`，逐位移位后可得到 `KXRKGI`。同一明文字母在不同位置可能加不同偏移，因此不像单表替代那样固定映射。

弱点是密钥循环。攻击者若估计出密钥长度，就能把密文按周期分组，每组退化成单字母表替代，再做频率分析。

### Vernam 与一次一密

Vernam 密码使用按位异或：

```text
C_i = P_i xor K_i    # P_i 是第 i 位明文比特，K_i 是第 i 位密钥比特；异或后得到第 i 位密文 C_i
P_i = C_i xor K_i    # 接收方用同一 K_i 再异或一次；因为 X xor Y xor Y = X，所以恢复 P_i
```

`xor` 是按位异或，相同为 0，不同为 1。它可逆，因为 `P_i xor K_i xor K_i = P_i`。例：`P_i=1`，`K_i=0`，则 `C_i=1`；接收方再用 `1 xor 0` 得回 `1`。

一次一密要求密钥真正随机、与明文等长、只使用一次、只被通信双方知道。满足这些条件时可以达到无条件安全。缺点是密钥分发极难，密钥消耗量等于消息量。

### 置换密码、乘积密码与转子机

置换密码不改变字母，只改变位置。Rail Fence 是典型例子，把字母按多行写入，再按行读出。乘积密码把替代和置换组合多轮，使统计特征扩散到更多位置。现代分组密码中的混淆和扩散思想就来源于此。

转子机如 Enigma 通过多个转子的运动动态改变替代表。它说明动态、多轮、组合式替代能增加破解难度，但最终仍要经受密码分析。

### 易错点

密钥空间大不等于一定安全。单字母表替代有 `26!` 种密钥，却会被词频分析击破。

一次一密不是“密钥很长”就安全，必须真随机、等长、只用一次。重复使用同一密钥流会造成严重泄露。

传统密码的意义不在于今天直接使用，而在于理解混淆、扩散、密钥空间、统计分析和密码设计思想。

## 3. 第二章-对称加密.pptx

### 本章主线

对称加密中，通信双方使用同一密钥。它速度快，适合大量数据，但最大问题是密钥分发和密钥管理。若 `N` 个用户两两安全通信，需要 `N(N-1)/2` 个共享密钥，这在大规模系统中不可接受。因此现实协议常用公钥密码协商或分发会话密钥，再用对称密码加密数据。

### Feistel 结构

Feistel 网络把输入分成左右两半，每轮形式为：

```text
L_i = R_{i-1}                              # L_i 是第 i 轮左半块；它直接取上一轮右半块 R_{i-1}
R_i = L_{i-1} xor F(R_{i-1}, K_i)          # R_i 是第 i 轮右半块；把上一轮左半块 L_{i-1} 与轮函数输出异或
```

`K_i` 是本轮子密钥，`F` 是轮函数。新的左半边直接取上一轮右半边；新的右半边由上一轮左半边和轮函数输出异或得到。由于异或可逆，解密时只需子密钥逆序，轮函数 `F` 本身不必可逆。这是 DES 等分组密码的重要基础。

### DES

DES 是 64 bit 分组、有效密钥 56 bit 的分组密码。整体结构包括初始置换、16 轮 Feistel 迭代、左右交换和逆初始置换。

每轮 `F` 函数包括扩展置换、与子密钥异或、S 盒代换和 P 盒置换。S 盒是非线性核心。例如 6 bit 输入 `101001`，首尾两位 `11` 决定行，中间四位 `0100` 决定列，再到 S 盒表中查得 4 bit 输出。首尾取行、中间取列是 DES S 盒题常见考点。

DES 的问题是密钥过短。`2^56` 的穷举空间在现代计算条件下不再足够保护高价值数据。

### 3DES

3DES 通过多次 DES 增强安全性，典型结构是加密-解密-加密：

```text
C = E_{K3}(D_{K2}(E_{K1}(P)))    # P 先用 K1 做 DES 加密，再用 K2 做 DES 解密形式变换，最后用 K3 加密得到 C
```

这里中间的 `D` 不是为了还原明文，而是为了兼容单 DES。当密钥取特定相同值时，3DES 可退化成普通 DES。3DES 安全性高于 DES，但速度慢，现代系统更多使用 AES 或 SM4。

### 为什么不用 2DES

直接把 DES 加密两次看起来能把密钥长度从 56 bit 提升到 112 bit：

```text
C = E_{K2}(E_{K1}(P))    # P 先用 K1 加密得到中间值 X，再用 K2 加密得到最终密文 C
X = E_{K1}(P)            # X 是第一层 DES 输出的中间值；攻击者可从已知明文 P 向前枚举 K1 得到许多 X
X = D_{K2}(C)            # 攻击者也可从已知密文 C 向后枚举 K2 得到许多 X；两边相同的 X 形成“中间相遇”
```

这就是中间相遇攻击。攻击者不必穷举 `2^112` 个密钥组合，而是分别枚举 `K1` 和 `K2`，在中间值 `X` 处匹配，复杂度约降到 `2^57`，只比单 DES 略高。因此 2DES 不值得使用，实际采用 3DES 或直接使用 AES/SM4。

### AES

AES 是现代对称分组密码标准，分组长度 128 bit，密钥长度可为 128、192、256 bit。AES 状态可看作 4x4 字节矩阵。

每轮核心操作包括：

- SubBytes：字节代换，提供非线性。
- ShiftRows：行移位，使字节跨列扩散。
- MixColumns：列混合，使一列内字节相互影响。
- AddRoundKey：与轮密钥异或，把密钥混入状态。

例：某状态字节为 `0x57`，轮密钥字节为 `0x83`，AddRoundKey 后为 `0xD4`。这说明 AES 的密钥不是单独附加在最后，而是在每轮运算中反复参与。

### 分组密码工作模式

ECB 模式：每个明文块独立加密，`C_i = E_K(P_i)`。（注：`P_i` 是第 i 个明文块，`K` 是分组密码密钥，`C_i` 是对应密文块；每块互不影响。）相同明文块会产生相同密文块，容易泄露结构，不推荐用于普通数据加密。

CBC 模式：

```text
C_1 = E_K(P_1 xor IV)          # P_1 是第一块明文，IV 是初始化向量；先异或 IV，再用 K 加密成第一块密文 C_1
C_i = E_K(P_i xor C_{i-1})     # P_i 是第 i 块明文；先与上一块密文 C_{i-1} 异或，再加密得到 C_i
P_i = D_K(C_i) xor C_{i-1}     # 解密第 i 块时，先用 K 解开 C_i，再与上一块密文 C_{i-1} 异或恢复 P_i
```

这里上一块密文参与下一块加密，所以相同明文块在不同位置可能得到不同密文。`IV` 相当于第一块的“上一块密文”，应不可预测或至少不重复。

CTR 模式：

```text
KS_i = E_K(Nonce || Counter_i)    # Nonce 是一次性随机/唯一值，Counter_i 是第 i 个计数器；加密二者生成第 i 段密钥流 KS_i
C_i = P_i xor KS_i                # P_i 是第 i 段明文；与密钥流 KS_i 异或得到密文 C_i
```

它把分组密码变成密钥流生成器。`Nonce || Counter_i` 不能重复，否则会重复密钥流，导致两个密文异或后暴露两个明文的关系。

### 流密码与 RC4

流密码生成伪随机密钥流，再与明文异或：

```text
C_i = P_i xor KS_i    # P_i 是第 i 位/字节明文，KS_i 是第 i 位/字节密钥流；异或得到 C_i
P_i = C_i xor KS_i    # 接收方用同一密钥流 KS_i 再异或一次，恢复 P_i
```

`KS_i` 是第 `i` 位或第 `i` 字节密钥流。流密码的最大禁忌是密钥流重复。RC4 曾广泛用于 WEP、TLS 等场景，但由于存在偏差和使用问题，现代安全协议已基本不推荐使用。

### 易错点

对称加密速度快，但不解决“第一次怎么安全分发密钥”的问题。

ECB 不安全的原因不是算法本身坏，而是模式泄露重复明文结构。

IV、nonce、salt 都常被称为随机值，但用途不同：IV 用于加密随机化，nonce 用于唯一性或新鲜性，salt 用于防预计算字典。

## 4. 第二章-非对称加密.pptx

### 本章主线

非对称加密让每个用户拥有一对密钥：公钥公开，私钥保密。公钥可用于加密或验证签名，私钥可用于解密或签名。它解决了对称密码的密钥分发难题，但运算慢，因此常用于密钥交换、数字签名和混合加密。

### 公钥加密与数字签名

保密通信：

```text
A -> B: E_{PU_B}(M)          # A 发给 B；PU_B 是 B 的公钥，M 是消息；A 用 B 的公钥加密，使只有 B 能读
B: D_{PR_B}(C) = M           # B 解密；PR_B 是 B 的私钥，C 是收到的密文；B 用私钥恢复明文 M
```

A 使用 B 的公钥 `PU_B` 加密消息，只有 B 持有 `PR_B` 能解密。这提供保密性，但不能证明消息来自 A，因为任何人都能使用 B 的公钥加密。

数字签名：

```text
A -> B: M, Sign_{PR_A}(H(M))              # A 发送明文 M 和签名；PR_A 是 A 的私钥，H(M) 是消息摘要，签名证明 A 认可该消息
B: Verify_{PU_A}(H(M), signature)         # B 验证；PU_A 是 A 的公钥，B 用它检查签名是否对应当前消息摘要
```

A 用自己的私钥对消息摘要签名，B 用 A 的公钥验证。签名提供来源认证、完整性和抗抵赖，但不提供保密性。

### Diffie-Hellman 密钥交换

DH 让双方在不安全信道上协商共享秘密：

1. 双方公开大素数 `p` 和生成元 `g`。（注：`p,g` 是公开参数，不是秘密；所有参与方和攻击者都可以知道。）
2. Alice 选择私钥 `a`，计算 `A = g^a mod p`，发送 `A`。（注：`a` 是 Alice 私密随机数，不发送；`A` 是公开值，可在网络上传输。）
3. Bob 选择私钥 `b`，计算 `B = g^b mod p`，发送 `B`。（注：`b` 是 Bob 私密随机数，不发送；`B` 是公开值。）
4. Alice 计算 `K = B^a mod p`。（注：Alice 用 Bob 的公开值 `B` 和自己的私钥 `a` 计算共享密钥 `K`。）
5. Bob 计算 `K = A^b mod p`。（注：Bob 用 Alice 的公开值 `A` 和自己的私钥 `b` 计算同一个共享密钥 `K`。）
6. 因为 `(g^b)^a = (g^a)^b = g^{ab}`，双方得到相同密钥。（注：双方没有发送私钥，却都得到 `g^{ab} mod p`。）

小写 `a/b` 是保密随机数，大写 `A/B` 是公开值。攻击者看到 `p,g,A,B`，若要得到 `K`，需要从 `g^a mod p = A` 反推 `a`，这就是离散对数问题。

基本 DH 不认证身份，容易被中间人攻击。实际协议会把 DH 与证书、签名或预共享密钥结合。

### RSA

RSA 基于大整数分解困难。密钥生成：

1. 选择大素数 `p,q`。
2. 计算 `n = p*q`。
3. 计算 `phi(n) = (p-1)(q-1)`。
4. 选择 `e`，满足 `gcd(e, phi(n)) = 1`。
5. 求 `d`，满足 `e*d ≡ 1 mod phi(n)`。（注：`d` 是私钥指数，是 `e` 在模 `phi(n)` 意义下的乘法逆元。）
6. 公钥为 `(e,n)`，私钥为 `(d,n)`。

加密和解密：

```text
C = M^e mod n    # M 是明文数字，(e,n) 是接收方公钥；发送方用公钥指数 e 加密得到密文 C
M = C^d mod n    # C 是密文，(d,n) 是接收方私钥；接收方用私钥指数 d 解密恢复 M
```

若这是 B 的 RSA 密钥，则 A 用 B 的公钥指数 `e` 加密，B 用自己的私钥指数 `d` 解密。真实 RSA 不能裸用，需要填充；速度较慢，通常只加密会话密钥或签名摘要。

### ECC

ECC 基于椭圆曲线离散对数问题。曲线形式可写为：

```text
y^2 = x^3 + ax + b    # x,y 是曲线上点的坐标，a,b 是曲线参数；ECC 安全性来自点乘逆向困难
```

实际使用中，私钥是整数 `d`，公钥是点 `Q = dG`。知道 `d` 和基点 `G` 容易算出 `Q`，但只知道 `G` 和 `Q` 很难反推出 `d`。ECC 的优点是较短密钥达到较高安全强度，适合移动设备和高性能场景。国密 SM2 属于椭圆曲线公钥算法。

### 混合加密

典型流程：

1. 发送方生成随机会话密钥 `K_s`。（注：`K_s` 只用于本次会话，适合加密大量数据。）
2. 用 `K_s` 加密大量数据：`C = E_{K_s}(M)`。（注：`M` 是正文消息，`C` 是正文密文；对称加密速度快。）
3. 用接收方公钥加密 `K_s`：`E_{PU_B}(K_s)`。（注：`PU_B` 是 B 的公钥；这一步把会话密钥装进只有 B 能打开的“数字信封”。）
4. 接收方用私钥 `PR_B` 解出 `K_s`，再用 `K_s` 解密数据。（注：`PR_B` 只有 B 持有；B 先取出会话密钥，再解开正文密文。）

最终发送内容通常是两部分：加密数据 `C` 和被公钥保护的会话密钥。TLS、PGP、S/MIME、SET 都体现这一思想。

### 易错点

“用公钥加密”解决保密，不解决发送者身份。

“用私钥签名”解决认证、完整性、抗抵赖，不解决保密。

公钥密码不是替代对称密码，而是与对称密码协同：公钥用于认证和密钥交换，对称密钥用于大量数据传输。

## 5. 第三章-报文鉴别与哈希函数.pptx

### 本章主线

本章连接密码学与安全协议：如何确认消息没有被改，如何确认消息来自谁，为什么哈希、MAC 和数字签名提供的安全服务不同。

### 报文鉴别与身份认证

报文鉴别关注某一条消息是否完整、是否来自声称发送者。身份认证关注当前交互实体是否真是它声称的身份。前者可以是静态值，如 MAC 或签名；后者通常是动态协议，需要 nonce、时间戳或挑战应答防止重放。

### MAC

MAC 是带密钥的报文摘要：

```text
tag = MAC_K(M)                  # M 是报文，K 是 A/B 共享密钥；A 用 K 为 M 生成鉴别码 tag
A -> B: M || tag                # A 把原报文 M 和鉴别码 tag 一起发给 B；|| 表示字段连接
B: 重新计算 MAC_K(M) 并比较     # B 用同一密钥 K 重新计算 tag；若一致，说明 M 未被不知道 K 的人篡改
```

A 和 B 共享密钥 `K`。A 对消息计算 tag，B 收到后用同一密钥重新计算。如果一致，说明不知道 `K` 的攻击者难以篡改消息并伪造 tag。MAC 提供完整性和数据源认证，但不能抗抵赖，因为 B 也知道 `K`，第三方无法判断 tag 到底是谁生成的。

### CBC-MAC 与 DAA

PPT 中讲到用 DES-CBC 处理报文，取最后一个密文块作为 MAC：

```text
C_0 = 0                         # C_0 是初始值，CBC-MAC 中通常设为 0
C_i = E_K(P_i xor C_{i-1})      # P_i 是第 i 个报文分组；先与上一密文块 C_{i-1} 异或，再用 K 加密得到 C_i
MAC = C_n                       # C_n 是最后一个密文块；它受所有分组影响，因此作为整条报文的 MAC
```

最后一块密文受前面所有分组影响，因此可作为整个报文的压缩鉴别值。早期 DAA 基于 DES-CBC，但 DES 分组和密钥强度有限，MAC 太短时易被攻击。

### HMAC 与长度扩展攻击

不要把 `H(K || M)` 简单当成安全 MAC。许多 Merkle-Damgard 结构的哈希函数存在长度扩展特性：攻击者知道 `H(K || M)` 后，虽然不知道 `K`，仍可能构造 `M || padding || M2` 的合法哈希。这会让“密钥前缀哈希”在某些场景下被伪造。

HMAC 用双层结构避免这个问题：

```text
inner = H((K xor ipad) || M)                         # K 是共享密钥，ipad 是固定内层填充值；先把密钥和消息放入内层哈希
HMAC(K, M) = H((K xor opad) || inner)                # opad 是固定外层填充值；外层再把密钥混入一次，得到最终 HMAC
```

直观理解：攻击者即使看到最终 HMAC，也拿不到内层输入，更不能从一个可见哈希状态直接追加内容。实际协议中常用 HMAC-SHA256、HMAC-SHA1 等组合。学习时只需抓住一点：HMAC 是标准 MAC 构造，比简单 `H(K||M)` 更可靠。

### 哈希函数

哈希函数把任意长度输入映射为固定长度摘要：

```text
h = H(M)    # M 是任意长度消息，H 是哈希函数；输出固定长度摘要 h，作为消息指纹
```

哈希没有密钥，公开可算。它适合做消息指纹，但单独不能抵抗恶意替换。例如攻击者若同时替换软件下载文件和网页上的哈希值，用户仍会被骗。因此安全发布通常需要数字签名或可信渠道。

安全哈希应具有单向性、弱抗碰撞和强抗碰撞。MD5 和 SHA-1 已不适合碰撞安全场景；SHA-2、SHA-3、SM3 更常用于现代系统。

### 生日攻击

若哈希输出长度为 `n` bit，输出空间为 `N=2^n`，寻找任意碰撞的复杂度约为：

```text
sqrt(N) = 2^{n/2}    # N=2^n 是 n bit 哈希的输出空间；寻找任意碰撞约需输出空间平方根数量的尝试
```

这不是说能快速找到某个指定消息的前像，而是说随机制造许多消息时，约 `2^{n/2}` 次尝试就容易出现任意两条消息哈希相同。因此 128 bit 哈希的碰撞强度约 64 bit。

### 数字签名与哈希结合

实际签名通常签摘要：

```text
h = H(M)                  # M 是消息，H 是哈希函数；A 先把消息压缩成固定长度摘要 h
s = Sign_{PR_A}(h)        # PR_A 是 A 的私钥；A 对摘要 h 签名得到签名值 s
发送 M || s               # A 发送消息 M 和签名 s；接收方用 A 的公钥验证 s 是否匹配 M
```

接收方重新计算 `H(M)`，再用 A 的公钥验证签名。这样做效率高，并且消息任意一位改变都会导致摘要不同。签名提供完整性、来源认证和抗抵赖。

### 易错点

哈希没有密钥，不能单独证明消息来自谁。

MAC 有共享密钥，能证明“来自知道密钥的一方”，但不能抗抵赖。

数字签名使用私钥签、公钥验，可以抗抵赖，但需要 PKI 解决“公钥到底属于谁”。

## 6. 第四章-公开密钥基础设施PKI及其应用.pptx

### 本章主线

公钥密码解决了“公钥可以公开”的问题，但没有解决“这个公钥是谁的”。PKI 的核心任务是把身份和公钥可信绑定起来。数字证书就是这种绑定关系的载体，CA 是背书者。

### 数字证书结构

简化证书包含主体身份、主体公钥、颁发者、有效期、用途、算法和 CA 签名。CA 签名可写成：

```text
signature = Sign_{PR_CA}(H(证书主体内容))    # PR_CA 是 CA 私钥；CA 对证书内容摘要签名，生成证书签名 signature
```

CA 不是加密证书内容，而是用自己的私钥对证书内容摘要签名。用户用 CA 公钥验证签名。如果攻击者替换证书中的公钥，证书主体内容改变，签名验证会失败。

### X.509 v3 证书字段

X.509 v3 是常见公钥证书格式。学习时不必背全部编码细节，但要知道证书大致包含哪些信息：

- 版本号和序列号：标识证书格式和 CA 内部唯一编号。
- 签名算法：说明 CA 用什么算法签名，如 RSA/SHA-256。
- 颁发者：签发该证书的 CA 名称。
- 有效期：not before 和 not after，证书只在该时间窗口内有效。
- 主体名称：证书持有者，如域名、组织或个人。
- 主体公钥信息：主体的公钥、算法和参数。
- 扩展字段：密钥用途、扩展密钥用途、CRL/OCSP 地址、名称约束、策略等。
- 签名值：CA 对上述证书主体内容的签名。

验证 HTTPS 证书时，浏览器不仅检查签名，还会检查域名是否匹配、证书是否过期、用途是否允许服务器认证、证书是否被撤销。

### PKI 组成

CA 负责签发证书和撤销列表。RA 负责验证申请者身份，不直接签发证书。证书库或目录发布证书和 CRL。CRL 和 OCSP 用于查询证书是否已撤销。客户端软件如浏览器负责证书验证。

### 证书签发与发布流程

典型流程：

1. EE 向 RA 提交证书申请，包含身份材料和公钥。（注：`EE` 是最终实体，如用户或服务器；这一步提交“我是谁”和“我的公钥”。）
2. RA 验证 EE 身份，确认公钥归属。（注：`RA` 是注册机构；它不签证书，负责防止攻击者冒名申请。）
3. RA 向 CA 请求签发证书。（注：`CA` 是证书授权中心；RA 把已核验的申请交给 CA。）
4. CA 用 `PR_CA` 对证书内容签名。（注：`PR_CA` 是 CA 私钥；签名把 EE 身份和 EE 公钥绑定起来。）
5. CA/RA 将证书和撤销信息发布到目录。（注：目录用于让其他用户获取证书和 CRL/撤销状态。）
6. 其他用户查询或下载证书。（注：用户拿到证书后，还必须验证 CA 签名、有效期、用途和撤销状态。）

这里 EE 是最终实体，可以是用户、服务器或设备。RA 的作用是避免攻击者拿自己的公钥冒充别人。CA 的签名使别人能验证“身份-公钥绑定”。

### 证书链验证

层次 PKI 中，用户预先信任根 CA。验证目标证书时，从根 CA 到中间 CA 再到目标证书逐级验证。

例：A 信任根 CA，D 的证书由 CA21 签发，CA21 由 CA2 签发，CA2 由根 CA 签发。A 验证 D 时：

1. 用根 CA 公钥验证 CA2 证书。（注：根 CA 是 A 预先信任的信任锚；验证通过后，CA2 公钥才可信。）
2. 从 CA2 证书中取出 CA2 公钥。（注：这一步得到下一层验证所需的公钥。）
3. 用 CA2 公钥验证 CA21 证书。（注：CA2 为 CA21 背书，验证通过才信任 CA21 公钥。）
4. 从 CA21 证书中取出 CA21 公钥。（注：这把公钥用于验证最终主体 D 的证书。）
5. 用 CA21 公钥验证 D 的证书。（注：若签名正确，说明 D 的身份和公钥绑定未被篡改。）
6. 检查有效期、用途、域名、撤销状态和策略约束。（注：签名正确不代表证书当前一定可用，还要检查这些限制。）
7. 全部通过后，A 才信任 D 的公钥。（注：此后 A 才能用 D 公钥加密或验证 D 的签名。）

关键思想是：前一个可信公钥验证后一个证书签名，信任逐级传递。

### 证书撤销

证书未过期也可能失效，如私钥泄露、身份变化、权限取消。CRL 是 CA 定期发布的撤销列表，OCSP 是在线查询某张证书状态。浏览器访问 HTTPS 时通常会检查证书链、域名、有效期、用途和撤销状态。

### 多 CA 模型

单 CA 模型简单但不现实。层次模型通过根 CA 和中间 CA 建立信任路径。交叉认证让不同 CA 域互相信任。桥 CA 不直接给用户发证，而是连接不同 PKI 域，适合大型组织或跨域信任。

### PKI 应用

HTTPS/TLS 网站认证、邮件加密和签名、VPN、代码签名、智能卡和 USB Key 都依赖 PKI。PKCS 系列规范覆盖 RSA、DH、证书请求、私钥格式、密码令牌接口、个人证书包等内容。

常见 PKCS 可以这样理解：

- PKCS #1：RSA 加密和签名相关格式。
- PKCS #3：Diffie-Hellman 密钥协商。
- PKCS #5：基于口令的加密。
- PKCS #7：密码消息语法，用于签名/加密消息封装。
- PKCS #8：私钥信息格式。
- PKCS #10：证书请求格式，申请证书时常见。
- PKCS #11：密码令牌接口，如智能卡、USB Key、HSM。
- PKCS #12：个人证书和私钥打包格式，常见扩展名如 `.p12`、`.pfx`。

### 易错点

证书不需要保密，因为公钥本来公开；证书需要防篡改，因为攻击者最想替换的是公钥。

信任证书不是因为“证书看起来像真的”，而是因为证书链能验证到本地信任根。

CA 签名的是证书内容，不是用户私钥；用户私钥应该始终由用户自己保护。

## 7. 第五章 身份认证.pptx

### 本章主线

身份认证是一个交互过程，目标是让验证方确认宣称者身份。它最关注重放攻击，所以必须引入新鲜性：nonce、时间戳、一次性口令或会话密钥。

### 认证因子

认证因子包括所知道的东西，如口令、PIN；所拥有的东西，如门禁卡、U 盾、动态口令牌；所固有的东西，如指纹、人脸。双因素认证要求来自不同类别的证据，而不是两个口令。

### 口令与 Salt

系统不应保存明文口令，而应保存哈希。基本流程：

```text
注册：保存 H(password)                  # password 是用户口令；系统保存其哈希值，不保存明文口令
登录：计算 H(password')，与保存值比较    # password' 是本次输入；若哈希相同，说明输入口令与注册口令一致
```

即使保存哈希，也要防止离线字典攻击。Salt 的作用是让同一口令在不同用户或不同系统中得到不同哈希，增加预计算字典和彩虹表成本。Salt 不是秘密，它的价值在于唯一性和扩大攻击成本。

早期 UNIX 口令机制使用改造 DES 和 salt，后来把真实口令哈希从 `/etc/passwd` 移到权限更高的 `/etc/shadow`。

### 动态口令与 Lamport 哈希链

Lamport 方案：

1. 用户选择秘密 `w`。（注：`w` 是哈希链种子，只应由用户掌握。）
2. 计算 `w, h(w), h(h(w)), ..., h^t(w)`。（注：`h^t(w)` 表示对 `w` 连续哈希 t 次。）
3. 服务器初始保存 `h^t(w)`。（注：服务器保存链尾，不保存 `w`。）
4. 第一次登录提交 `h^{t-1}(w)`。（注：用户提交链尾的前一个值，作为一次性口令。）
5. 服务器哈希一次并与保存值比较。（注：若 `h(h^{t-1}(w)) = h^t(w)`，说明口令正确。）
6. 通过后，服务器把保存值更新为 `h^{t-1}(w)`。（注：旧值作废，下一次必须提交更前一个链值，防止重放。）

攻击者即使截获本次口令，也无法反推出下一次要提交的更早前像。缺点是使用次数有限，且不能自动认证服务器，仍可能被实时钓鱼。

### 质询-应答

基本流程：

```text
B -> A: rB                         # B 是验证方，rB 是 B 生成的一次性随机挑战，用于保证本轮新鲜性
A -> B: response = f(secret, rB)    # A 用秘密 secret 和挑战 rB 计算响应；response 必须随 rB 改变
B: 验证 response                    # B 用自己掌握的 secret 或验证材料检查 response，确认 A 知道秘密
```

`rB` 是本轮随机挑战，只使用一次。响应必须绑定秘密和本轮挑战。若攻击者重放旧响应，由于 `rB` 已不同，验证失败。

对称加密单向认证：

```text
B -> A: rB             # B 发出随机挑战 rB；攻击者即使截获旧响应，也无法用于新的 rB
A -> B: E_K(rB, B)     # A 用共享密钥 K 加密 rB 和接收方身份 B；证明 A 知道 K 且响应的是 B 的本轮挑战
```

A 用共享密钥加密 B 的挑战和 B 的身份。B 解密后看到自己刚发出的 `rB`，说明对方知道密钥且响应的是本轮挑战。加入 `B` 是为了绑定方向，防止反射攻击。

### 基于 MAC 和签名的认证

MAC 双向认证可写成：

```text
B -> A: rB                         # B 先发随机数 rB，要求 A 对本轮挑战作答
A -> B: rA, MAC_K(rA, rB, B)        # A 生成随机数 rA，并用共享密钥 K 对 rA、rB、B 生成 MAC，证明 A 知道 K
B -> A: MAC_K(rA, rB, A)            # B 用同一 K 对 rA、rB、A 生成 MAC，证明 B 也知道 K 并回应 A 的挑战
```

第二条证明 A 知道密钥并回应 B 的 nonce；第三条证明 B 知道同一密钥并回应 A 的 nonce。身份标识用于防反射。

签名认证：

```text
B -> A: rB                                      # B 发送一次性随机挑战 rB，防止 A 重放旧签名
A -> B: Cert_A, rA, B, Sign_{PR_A}(rA, rB, B)   # Cert_A 是 A 的证书，rA 是 A 的随机数，PR_A 是 A 私钥；签名绑定双方随机数和接收方 B
B: 验证 Cert_A，再用 PU_A 验证签名              # B 先从证书链确认 PU_A 属于 A，再用 PU_A 验证签名是否由 A 私钥生成
```

`Cert_A` 用于让 B 获得可信的 `PU_A`。签名覆盖双方随机数和接收方身份，证明 A 拥有私钥且正在参与本轮协议。

### Needham-Schroeder 对称密钥协议

目的：可信第三方 Trent/KDC 为 A 和 B 分发会话密钥 `K`。

前提：A 与 Trent 共享 `K_AT`，B 与 Trent 共享 `K_BT`。

流程：

```text
1. A -> T: A, B, N_A                                  # A 请求可信第三方 T 为 A 和 B 分配密钥；N_A 是 A 的随机数，用于绑定本次请求
2. T -> A: E_{K_AT}(N_A, B, K, E_{K_BT}(A, K))          # K_AT 是 A/T 长期密钥；T 把会话密钥 K 和给 B 的票据一起加密给 A
3. A -> B: E_{K_BT}(A, K)                              # K_BT 是 B/T 长期密钥；这是 T 写给 B 的票据，A 只能转发，不能解开
4. B -> A: E_K(B, N_B)                                 # B 解出票据得到 K 后，用会话密钥 K 加密自己的身份 B 和随机数 N_B 发给 A
5. A -> B: E_K(A, N_B - 1)                             # A 用会话密钥 K 回答 N_B-1，证明 A 也持有同一会话密钥 K
```

第 2 条外层只有 A 能解开，因此 A 知道这是 Trent 对本次 `N_A` 的回应；内层票据只有 B 能解开，A 只能转发。第 4、5 条用新会话密钥做双向质询，证明双方都得到 `K`。

旧版本问题是票据缺少鲜活性，攻击者可能重放旧票据。改进方法是加入时间戳或更严格的新鲜性检查。

### Kerberos

Kerberos 是基于 KDC 和对称密钥的认证系统，适合分布式 C/S 环境。它将认证功能从应用服务器中抽出，由 AS 和 TGS 集中处理。

第一阶段：用户向 AS 获取访问 TGS 的票。

```text
C -> AS: ID_C, ID_TGS, TS1                                      # C 是客户端，AS 是认证服务器；ID_C 是用户身份，ID_TGS 是票据服务器身份，TS1 是时间戳
AS -> C: E_{K_C}(K_C,TGS, ID_TGS, TS2, Lifetime, Ticket_TGS)      # K_C 由用户口令导出；AS 加密返回 C/TGS 会话密钥 K_C,TGS、有效期和 TGS 票据
Ticket_TGS = E_{K_TGS}(K_C,TGS, ID_C, AD_C, ID_TGS, TS2, Lifetime) # K_TGS 是 AS/TGS 共享密钥；Ticket_TGS 只有 TGS 能解开，客户端只能转交
```

`K_C` 由用户口令导出，只有用户能解开第一部分。`Ticket_TGS` 是给 TGS 的票据，客户端拿到但不能修改。

第二阶段：用户向 TGS 申请访问服务器 V 的票。

```text
C -> TGS: ID_V, Ticket_TGS, Authenticator_C                       # C 请求访问服务器 V；提交 TGS 票据和认证子证明自己
Authenticator_C = E_{K_C,TGS}(ID_C, AD_C, TS3)                    # K_C,TGS 是 C/TGS 会话密钥；认证子包含用户身份、客户端地址 AD_C 和时间戳 TS3
TGS -> C: E_{K_C,TGS}(K_C,V, ID_V, TS4, Ticket_V)                 # TGS 返回 C/V 会话密钥 K_C,V 和服务器票据 Ticket_V，并用 K_C,TGS 加密给 C
Ticket_V = E_{K_V}(K_C,V, ID_C, AD_C, ID_V, TS4, Lifetime)         # K_V 是 TGS/V 共享密钥；Ticket_V 只有服务器 V 能解开
```

认证子证明客户端当前持有会话密钥且请求新鲜。`Ticket_V` 只有服务器 V 能解开。

第三阶段：访问服务器。

```text
C -> V: Ticket_V, Authenticator'_C                 # C 把服务器票据和新的认证子交给 V，申请访问服务
Authenticator'_C = E_{K_C,V}(ID_C, AD_C, TS5)      # K_C,V 是 C/V 会话密钥；认证子用新时间戳 TS5 证明请求不是重放
V -> C: E_{K_C,V}(TS5 + 1)                         # V 用同一 K_C,V 返回 TS5+1，证明 V 也拿到了会话密钥，实现双向认证
```

V 解开票据得到 `K_C,V`，再解开认证子确认客户端身份和时间戳。最后返回 `TS5+1`，证明 V 也知道该会话密钥，实现双向认证。

### 易错点

认证不是“发一个口令”这么简单，而是要防重放。

Ticket 是给服务器看的，客户端通常看不懂、改不了。

Kerberos 依赖时间戳，主机时间不同步会导致认证失败。

## 8. 第六章 Web&EC Security.pptx

### 本章主线

Web 安全关注浏览器、服务器、网络传输和电子商务多方交易。HTTP 本身明文传输，不提供保密性、完整性和服务器认证。TLS/SSL 为 Web 传输提供安全通道，SET 则面向信用卡支付场景提供多方安全交易机制。

### Web 威胁

Web 服务器通常对外开放，容易成为攻击入口。威胁包括钓鱼、会话劫持、XSS、SQL 注入、参数篡改、弱配置、明文传输和客户端木马。TLS 只能保护网络传输阶段，不能解决服务器漏洞或客户端已中木马的问题。

### TLS/SSL 位置与目标

TLS 位于 TCP 之上、应用层之下。HTTPS 就是 HTTP over TLS。TLS 提供机密性、完整性、服务器认证、可选客户端认证、算法协商和相对高效的数据传输。

TLS 由记录协议、握手协议、密码变更协议和警告协议组成。握手协议负责协商算法、交换随机数、认证服务器、协商密钥；记录协议负责分段、加密、完整性保护和封装应用数据。

### TLS 握手流程

第一阶段 Hello：

```text
Client -> Server: ClientHello    # Client 是浏览器/客户端；ClientHello 发送支持的 TLS 版本、随机数、会话 ID 和 CipherSuite 列表
Server -> Client: ServerHello    # Server 是服务器；ServerHello 选择版本、CipherSuite，并返回服务器随机数
```

双方协商版本、随机数、会话 ID、CipherSuite 和压缩方法。随机数参与密钥派生，也具有挑战意义。

第二阶段服务器认证：

```text
Server -> Client: Certificate              # Certificate 是服务器证书；客户端用它获得服务器公钥并验证服务器身份
Server -> Client: ServerKeyExchange         # ServerKeyExchange 携带密钥交换参数；需要时服务器会对参数签名，证明拥有私钥
Server -> Client: CertificateRequest 可选   # CertificateRequest 表示服务器要求客户端也提供证书，用于客户端认证
Server -> Client: ServerHelloDone           # ServerHelloDone 表示服务器 Hello 阶段结束，轮到客户端继续握手
```

服务器发送证书。客户端验证证书链、域名、有效期、用途和撤销状态，并验证服务器对握手参数的签名，确认服务器拥有证书对应私钥。

第三阶段客户端密钥交换：

```text
Client -> Server: ClientKeyExchange          # ClientKeyExchange 携带客户端密钥交换材料，如 RSA 加密的 premaster_secret 或 DH 公共值
Client -> Server: CertificateVerify 可选     # CertificateVerify 是客户端用私钥对握手内容签名，证明自己拥有客户端证书对应私钥
```

传统 RSA 方式中，客户端生成 `premaster_secret`，用服务器公钥加密：

```text
Client -> Server: E_{PU_Server}(premaster_secret)                 # PU_Server 是服务器证书中的公钥；客户端用它加密预主密钥 premaster_secret
Server: D_{PR_Server}(...) = premaster_secret                     # PR_Server 是服务器私钥；只有真正服务器能解密得到 premaster_secret
master_secret = PRF(premaster_secret, ClientRandom, ServerRandom) # PRF 是密钥派生函数；双方用预主密钥和两个随机数派生主密钥 master_secret
```

只有真正持有服务器私钥的一方能解出预主密钥。随后双方用预主密钥和随机数派生记录协议使用的对称密钥。

第四阶段切换密码：

```text
Client -> Server: ChangeCipherSpec, Finished    # ChangeCipherSpec 表示客户端开始使用新密钥；Finished 校验前面握手是否被篡改
Server -> Client: ChangeCipherSpec, Finished    # 服务器同样切换到新密钥并发送 Finished；双方确认后开始保护应用数据
```

Finished 报文验证整个握手过程未被篡改。从此应用数据用协商好的对称密钥保护。

### TLS 会话恢复

完整 TLS 握手需要证书验证和密钥交换，成本较高。会话恢复用于让客户端和服务器在短时间内复用已有会话参数，减少握手开销。

典型思想如下：

```text
Client -> Server: ClientHello(session_id 或 ticket)    # 客户端带上以前会话的标识或会话票据，表示希望恢复
Server -> Client: ServerHello(确认恢复)                 # 服务器若认可该会话，就跳过部分完整握手步骤
双方基于旧 master_secret 派生新记录密钥                # 不直接重复使用旧记录密钥，而是结合新随机数派生本次连接密钥
```

会话恢复的重点是效率，不是降低安全要求。实现时仍要保证会话票据加密保护、有效期可控、密钥更新合理。若会话恢复参数泄露，攻击者可能恢复会话，因此服务器端 ticket key 和会话缓存也需要保护。

### SET 与双重数字签名

SET 面向信用卡支付，多方包括持卡人、商家、支付网关、发卡行、收单行和 CA。SET 与 SSL/TLS 的区别是：TLS 主要保护两方通道，SET 保护多方交易关系，并强调隐私隔离。

设 `PI` 是支付信息，`OI` 是订单信息：

```text
PIMD = H(PI)                # PI 是支付信息，PIMD 是支付信息摘要；商家可拿摘要但看不到支付明文
OIMD = H(OI)                # OI 是订单信息，OIMD 是订单信息摘要；银行可拿摘要但看不到订单明文
POMD = H(PIMD || OIMD)      # POMD 把支付摘要和订单摘要绑定，防止两部分被拆换
DS = Sign_{PR_C}(POMD)      # PR_C 是持卡人私钥；DS 是持卡人对绑定摘要的双重数字签名
```

持卡人签的不是支付明文或订单明文，而是两者摘要组合后的值。商家看到订单和支付摘要，银行看到支付和订单摘要，双方都能验证同一笔交易的绑定关系，但商家看不到支付明文，银行看不到订单明文。

支付信息通常用数字信封保护：

```text
E_{K_s}(PI, OIMD, DS)    # K_s 是一次性会话密钥；用它加密支付信息 PI、订单摘要 OIMD 和双重签名 DS
E_{PU_bank}(K_s)         # PU_bank 是银行/支付网关公钥；用它加密 K_s，只有银行能用私钥取出会话密钥
```

只有银行能用私钥解出 `K_s`，再解开支付信息。

### 易错点

HTTPS 不等于网站绝对安全，它保护传输，不修复服务器漏洞和用户终端恶意软件。

证书验证不只是看有没有锁标志，还要验证证书链、域名、有效期、用途和撤销状态。

SET 双重签名的关键不是“双重加密”，而是把订单和支付摘要绑定，同时隔离双方隐私。

## 9. 第七章 TCPIP协议及其安全隐患.pptx

### 本章主线

TCP/IP 协议族的很多安全问题来自早期设计假设：默认网络参与者较可信，协议更关注连通性和效率，而不是认证和防伪。DNS、ARP、DHCP、IP、TCP、UDP、HTTP 都存在不同层面的安全隐患。

### 分层地址

应用层使用域名，传输层使用端口，网络层使用 IP 地址，数据链路层使用 MAC 地址，物理层传输比特。DNS 完成域名到 IP 的映射，ARP 完成 IP 到 MAC 的映射。攻击常发生在这些映射环节。

### DNS 安全

DNS 将主机名映射到 IP 地址。根域名服务器、顶级域名服务器、权限域名服务器和本地域名服务器共同完成查询。本地域名服务器通常缓存结果。

DNS 威胁包括：

- 直接攻击 DNS 服务器，使其无法服务。
- 缓存中毒，使服务器缓存错误记录。
- 域名劫持，改变域名解析控制权。
- 反射放大攻击，利用 DNS 响应流量攻击第三方。

DNSSEC 使用数字签名保护 DNS 记录真实性和完整性，但不负责加密查询内容。

### IPsec 协议体系

IPsec 在网络层保护 IP 报文，可以用于主机到主机、主机到网关或网关到网关的安全通信。与 TLS 保护某个应用连接不同，IPsec 更靠近网络层，常用于 VPN 和站点互联。

两个核心子协议：

- AH：Authentication Header，提供数据源认证、完整性和抗重放，但不加密载荷，因此不提供保密性。
- ESP：Encapsulating Security Payload，可提供加密保密性，也可提供完整性、认证和抗重放，是实际中更常用的形式。

两种工作模式：

- 传输模式：保护 IP 载荷，原 IP 头仍保留，常用于主机到主机。
- 隧道模式：把整个原 IP 包封装进新的 IP 包，常用于网关到网关 VPN。

IPsec 依赖安全关联 SA 描述一条单向安全通信的参数，包括算法、密钥、SPI、生命周期等。实际部署中通常由 IKE 自动协商密钥和 SA。复习时抓住对比即可：`AH` 重完整性和认证，`ESP` 可加密；`传输模式` 保留原 IP 头，`隧道模式` 封装整个原 IP 包。

### ARP 安全

正常 ARP 流程：

1. A 要给 B 发帧，但不知道 B 的 MAC。（注：`A/B` 是局域网主机，`MAC` 是数据链路层地址。）
2. A 广播 ARP 请求：“谁是 IP_B？”（注：`IP_B` 是 B 的 IP 地址；广播让同一局域网所有主机都能收到。）
3. B 单播 ARP 响应：“IP_B 的 MAC 是 MAC_B。”（注：`MAC_B` 是 B 的网卡地址；只有 B 应该对自己的 IP 作答。）
4. A 将 `IP_B -> MAC_B` 写入 ARP 缓存。（注：后续发往 `IP_B` 的帧会使用 `MAC_B` 作为目的 MAC。）

ARP 缺陷是缺少认证。主机可能接受伪造响应，导致缓存污染。

ARP 欺骗中间人流程：

```text
攻击者 -> 目标主机: 网关 IP 对应 MAC_Attacker    # MAC_Attacker 是攻击者网卡地址；目标误以为攻击者就是网关
攻击者 -> 网关: 目标 IP 对应 MAC_Attacker        # 网关误以为攻击者就是目标主机
目标主机 -> 攻击者 -> 网关                       # 目标发往外网的流量先到攻击者，再由攻击者转发给网关
网关 -> 攻击者 -> 目标主机                       # 返回流量也先到攻击者，攻击者形成双向中间人
```

攻击者若转发流量，双方可能无感；若篡改，可破坏完整性；若丢弃，可造成拒绝服务。

### DHCP 安全

DHCP 自动分配 IP、网关、DNS 等参数。问题是客户端通常不认证服务器。伪造 DHCP 服务器可向客户端下发攻击者控制的网关或 DNS，使后续流量经过攻击者。

防御包括 DHCP Snooping、可信端口、固定关键服务器地址和网络准入控制。

### IP/TCP/UDP/HTTP 安全

IP 层问题包括源地址欺骗、源路由欺骗和分片攻击。

TCP 三次握手：

```text
Client -> Server: SYN        # SYN 是客户端发起连接请求，表示希望建立 TCP 连接
Server -> Client: SYN/ACK    # SYN/ACK 表示服务器同意建立连接，并确认收到客户端 SYN
Client -> Server: ACK        # ACK 表示客户端确认服务器 SYN/ACK，三次握手完成
```

SYN Flood 中，攻击者发送大量 SYN，使服务器分配半连接资源，但不完成 ACK，最终耗尽队列。防御包括 SYN Cookie、缩短超时、增大队列、限速和流量清洗。

UDP 无连接，容易被伪造源地址并用于泛洪或反射放大。HTTP 原生明文，容易遭受窃听、会话劫持、参数篡改、钓鱼和 XSS。

### 易错点

DNSSEC 解决记录伪造和篡改，不解决查询内容隐私。

ARP 欺骗通常是局域网攻击，但一旦成功可支撑 DNS 欺骗、会话劫持和中间人攻击。

SYN Flood 攻击目标是可用性，不是破解密码。

## 10. 第七章 网络攻击.pptx

### 本章主线

网络攻击不是单个动作，而是从信息收集、扫描、漏洞发现、渗透、控制、隐蔽到扩展的链条。理解攻击流程有助于理解防御部署为什么需要分层。

### 攻击生命周期

典型过程：

```text
信息收集 -> 扫描 -> 漏洞分析 -> 渗透 -> 权限提升 -> 隐蔽和扩展    # 从公开情报到获取权限再到长期控制，是典型攻击链
```

信息收集包括域名、IP、组织架构、员工邮箱、公开服务。扫描包括 Ping、Tracert、Nslookup、Whois、端口扫描和漏洞扫描。渗透后攻击者可能安装后门、清除日志、横向移动或控制肉鸡。

### 扫描工具

Ping 用于检测主机是否可达。Tracert 用于观察路由路径。Nslookup 查询 DNS。Whois 查询域名注册信息。Nmap 和 SuperScan 用于端口扫描。Nessus、X-Scan、Retina、LANguard 等用于漏洞扫描。

防御角度要最小化开放服务，隐藏不必要版本信息，及时修补漏洞，对扫描行为做 IDS/IPS 检测。

### 嗅探与 MAC 欺骗

共享式集线器环境中所有帧被广播，攻击者可直接嗅探。交换机环境中，攻击者可能通过 MAC 表溢出或 ARP 欺骗让流量经过自己。

MAC 地址欺骗是伪造源 MAC，使交换机学习错误转发表。防御包括端口安全、静态绑定和禁止未授权设备接入。

### ARP、DNS、DHCP、路由与 IP 欺骗

ARP 欺骗通过污染 ARP 缓存形成中间人。DNS 欺骗把域名解析到攻击者主机。DHCP 欺骗向客户端下发恶意网关或 DNS。路由欺骗改变报文转发路径。IP 欺骗伪造源地址，用于隐藏攻击者或配合反射攻击。

### DoS 与 DDoS

DoS 破坏可用性。常见方式包括 SYN Flood、UDP Flood、ACK Flood、DNS/NTP 反射放大和 Smurf 攻击。

Smurf 攻击流程：

1. 攻击者伪造受害者源 IP。（注：响应会被发给受害者，而不是攻击者。）
2. 向广播地址发送 ICMP 请求。（注：广播地址会让同一网络内多个主机收到请求。）
3. 大量主机同时回复受害者。（注：多个响应形成放大流量。）
4. 受害者带宽和处理能力被耗尽。（注：攻击目标是可用性。）

防御包括禁用定向广播、源地址过滤、流量清洗、Anycast/CDN、限速和应急预案。

### 课堂案例

Heartbleed 体现实现漏洞可导致内存信息泄露。即使 TLS 协议目标是安全的，具体软件实现错误仍可能破坏安全。

APT、震网、乌克兰断电等案例体现攻击组合化：钓鱼投递、漏洞利用、凭证窃取、横向移动、工控操作、日志破坏和拒绝服务可能同时出现。

### 易错点

端口扫描本身不一定是入侵成功，但它是攻击准备的重要步骤。

嗅探是被动攻击，ARP 欺骗是主动攻击；ARP 欺骗成功后可帮助攻击者嗅探。

DDoS 的难点在于攻击源分散、流量大、合法请求和恶意请求混杂。

## 11. 第八章 防火墙.pptx

### 本章主线

防火墙是边界访问控制设备，在内部网和外部网之间定义必经之点，根据安全策略允许或拒绝流量。它是重要防线，但不能解决所有问题。

### 防火墙目标与局限

防火墙目标包括服务控制、方向控制、用户控制、行为控制、日志审计、报警、NAT、计费和 IPSec 平台支持。

局限包括：不能防止绕过防火墙的连接，不能完全防止内部攻击，不能自动修复应用漏洞，不能识别所有恶意文件，不能看懂所有加密隧道中的恶意行为。

### 包过滤防火墙

包过滤根据源 IP、目的 IP、协议、端口和 TCP 标志位决定放行或丢弃。

例：允许内网访问外部 Web：

```text
出方向：源 IP=内网，目的端口=80，TCP，允许        # 允许内网主机主动访问外部 Web 服务
入方向：源端口=80，ACK=1，目的 IP=内网，允许       # 只允许外部 Web 服务器返回与已发起连接相关的响应包
默认：拒绝                                         # 未明确匹配允许规则的流量全部丢弃，属于白名单思路
```

无状态包过滤把每个包独立判断，难以准确识别复杂会话。它易受 IP 欺骗、源路由、小碎片和复杂协议配置问题影响。

### 有状态分组过滤

有状态防火墙维护连接状态。内网主机主动访问 Web 服务器后，防火墙记录五元组和状态，临时允许对应返回流量。会话结束或超时后删除状态。有状态过滤比纯包过滤准确，但仍不深入理解应用层语义。

### 应用层网关

应用层网关又称代理。它代表内网客户机访问外部服务器：

```text
内网客户机 -> 应用层网关        # 内网客户机不直接访问外部服务器，而是把请求交给代理/网关
应用层网关 -> 外部服务器        # 网关代表内网客户机向外部服务器发起请求
外部服务器 -> 应用层网关        # 外部服务器把响应返回给网关，通常看不到真实内网主机
应用层网关 -> 内网客户机        # 网关检查、记录后再把响应交还给内网客户机
```

外部服务器看到的是代理，而非真实内网主机。应用层网关可以检查应用内容、做用户认证和详细日志，但每种服务需要专门代理，性能开销较大。

### IDS/IPS 与防火墙

防火墙在线决定放行或丢弃。IDS 可旁路部署，分析长期、多连接、多主机行为并告警。IPS 在线部署，发现攻击后直接阻断。防火墙适合边界控制，IDS/IPS 适合发现和阻断攻击特征，二者应配合。

### 部署方案

包过滤路由器方案简单但防护层次少。屏蔽主机方案在包过滤后放堡垒主机。双宿主堡垒主机通过两个网卡隔离内外网。屏蔽子网方案使用外部路由器、DMZ、内部路由器形成多层防护，是常见部署方式。

### 易错点

防火墙不是万能安全设备。它只控制经过自己的流量，绕过路径、内部攻击、应用漏洞和客户端木马都可能绕过或隐藏于规则之外。

默认策略中，白名单比黑名单更保守。高安全边界通常采用“未明确允许即拒绝”。

工作层次越低效率越高但语义越弱；工作层次越高控制越细但开销越大。

## 12. 第九章 区块链.pptx

### 本章主线

区块链最初是为比特币记账系统设计的密码学和 P2P 解决方案。它用哈希链保证历史难篡改，用 Merkle 树保证区块内交易集合完整，用工作量证明和最长链原则实现去中心化共识，用数字签名证明交易授权。

### 区块和哈希链

区块是账本页，区块链是按顺序连接的账本页。区块头包含前一区块头哈希、Merkle 根、时间戳、难度和 nonce。

如果攻击者修改历史区块：

1. 该区块哈希改变。（注：区块内容一变，`H(区块头)` 就会改变。）
2. 后一区块头中的前一区块哈希对不上。（注：后一区块保存的 `prev_hash` 不再等于被改区块的新哈希。）
3. 后续所有区块都需要重算。（注：攻击者必须重新寻找每个后续区块的 nonce。）
4. 若没有足够算力，攻击链难以超过诚实链。（注：最长链/累计工作量规则会拒绝较弱分支。）

### Merkle 树

Merkle 树叶节点是交易哈希，父节点是两个子节点哈希连接后再哈希，根哈希写入区块头。验证某交易是否在区块中时，不需要下载所有交易，只需获取到根路径上的兄弟哈希，计算量约为 `O(log N)`。

任何交易改变都会逐级改变根哈希，因此 Merkle 根保证区块内交易集合完整性。

### 工作量证明

矿工寻找 nonce，使区块头哈希小于难度目标：

```text
H(prev_hash, nonce, Merkle_root, 其他头部字段) < E    # prev_hash 是前一区块哈希，nonce 是矿工反复尝试的数，Merkle_root 是交易集合摘要，E 是难度目标
```

`E` 是难度目标。目标越小，满足条件的哈希越少，挖矿越难。验证很容易，只需算一次哈希；寻找很难，只能大量尝试。

比特币约 10 分钟出一个区块，每 2016 个区块调整一次难度：

```text
新目标值 = 当前目标值 * 实际 2016 块耗时 / 理论 2016 块耗时    # 实际耗时短则目标值变小、难度升高；实际耗时长则目标值变大、难度降低
```

若最近出块太快，实际耗时小于理论耗时，新目标值变小，难度上升；若出块太慢，目标值变大，难度下降。

### 共识、确认与 51% 攻击

多个矿工可能同时挖出区块，网络会临时分叉。系统承认最长链或累计工作量最大链。交易刚进入一个区块时还不稳定，通常等待多个确认。PPT 中提到 6 个确认，用于降低被回滚风险。

51% 攻击指攻击者掌握超过诚实矿工总和的算力，可能构造更长链改写近期交易。但成本高，会破坏系统信任和经济价值。

### 交易签名

比特币交易包含输入、输出、付款人公钥和签名。验证交易时：

1. 根据付款人公钥生成地址。（注：公钥哈希可对应钱包地址，用于证明该公钥与上一输出匹配。）
2. 与上一笔交易输出地址比较。（注：若地址不匹配，说明付款人无权花费该输出。）
3. 用付款人公钥验证当前交易签名。（注：签名验证通过，说明交易由对应私钥持有者授权。）
4. 检查输入金额大于等于输出金额。（注：防止凭空创造金额，差额通常作为手续费。）

数字签名证明交易由私钥持有者发起，且交易内容未被篡改。比特币是伪匿名：地址不直接等于真实身份，但账本公开，地址一旦与现实身份关联，历史可被追踪。

### 易错点

区块链的不可篡改不是绝对不能改，而是篡改需要重新完成大量工作并追上主链。

PoW 的难点在寻找 nonce，验证却很容易。

公钥地址不等于完全匿名，公开账本会带来可追踪性。

## 13. 实验文档补充：ARP、DNS 与实验报告

### ARP 攻击实验

实验核心是观察 ARP 缓存污染和中间人位置形成。学习时重点看三个对象：目标主机、网关、攻击者。

典型步骤：

1. 查看目标主机和网关的正常 IP-MAC 映射。（注：记录攻击前 `网关 IP -> 网关 MAC` 和 `目标 IP -> 目标 MAC` 的正常状态。）
2. 攻击者向目标发送伪造 ARP 信息：“网关 IP 对应 MAC_Attacker”。（注：让目标把攻击者误认为网关。）
3. 攻击者向网关发送伪造 ARP 信息：“目标 IP 对应 MAC_Attacker”。（注：让网关把攻击者误认为目标主机。）
4. 目标和网关的双向流量经过攻击者。（注：攻击者位于通信路径中间，可以嗅探、篡改或丢弃流量。）
5. 若攻击者开启转发，通信可继续；若不开启转发，会造成断网。（注：转发决定攻击表现为隐蔽中间人还是拒绝服务。）

观察点：攻击前后目标主机 ARP 表中网关 IP 对应的 MAC 是否变为攻击者 MAC；攻击者是否能抓到目标流量。

### DNS 欺骗实验

DNS 欺骗常与 ARP 欺骗结合。攻击者先成为中间人，再伪造 DNS 响应，使受害者访问正常域名时被解析到攻击者指定 IP。

攻击链：

```text
接入同一局域网              # 攻击者需要能影响受害者二层/局域网通信
-> ARP 毒化成为中间人       # 先污染 ARP 缓存，让受害者流量经过攻击者
-> 截获 DNS 查询             # 攻击者看到受害者发出的域名解析请求
-> 返回伪造解析结果          # 攻击者抢先返回恶意 IP，使域名指向攻击者主机
-> 受害者访问伪造页面        # 受害者浏览器访问攻击者控制的站点，可能被钓鱼或投递恶意内容
```

防御包括 ARP 防护、可信 DNS、DNSSEC、HTTPS 证书校验、浏览器安全和用户不忽略证书错误。

### 实验报告写法建议

实验报告不要只截图工具界面，要解释“为什么”。例如 ARP 实验要写清楚协议缺陷、欺骗报文内容、ARP 表变化、流量路径变化和防御方法。DNS 实验要写清楚为什么仅伪造 DNS 响应还不够，为什么常需要先让攻击者处于中间人位置。

## 14. 阶段性学习路线

第一阶段先掌握安全目标和攻击分类：保密、完整性、可用性、认证、抗抵赖，被动攻击和主动攻击。

第二阶段掌握密码学基础：传统密码理解思路，对称加密理解效率和模式，非对称加密理解密钥分发和签名，哈希/MAC/签名理解安全服务差异。

第三阶段掌握信任和认证：PKI 解决公钥归属，身份认证协议解决实体身份和重放，Kerberos 解决分布式单点登录。

第四阶段进入网络协议和应用安全：TLS、SET、DNS、ARP、DHCP、TCP、HTTP、防火墙、IDS/IPS。

第五阶段理解综合攻击和前沿应用：网络攻击生命周期、实验中的 ARP/DNS 攻击、区块链中的哈希链、Merkle 树、PoW 和交易签名。

学习时最重要的能力是把“公式”和“人话流程”对应起来。例如 `E_{PU_B}(K_s)` 不是单纯一个符号，而是“发送方用 B 的公钥包住会话密钥，只有 B 能用私钥取出”；`Sign_{PR_A}(H(M))` 不是加密消息，而是“A 用私钥给消息摘要盖章，别人用 A 的公钥验章”。
