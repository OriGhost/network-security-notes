## 1. 第一章 网络安全引言

### 1.1 必会安全属性

网络安全基本属性常以判断、选择、多选出现。

| 属性 | 含义 | 常见机制 | 易错点 |
|---|---|---|---|
| 保密性 | 防止未授权主体获取信息 | 加密、访问控制、VPN | “加密”主要解决保密，不自动解决身份真实性 |
| 完整性 | 防止数据被未授权修改、插入、删除 | 哈希、MAC、数字签名 | 完整性关注“是否被改”，不是“是否被看见” |
| 可用性 | 授权用户需要时能使用服务 | 冗余、限速、抗 DoS、备份 | 拒绝服务攻击主要破坏可用性 |
| 可认证性 | 确认通信实体、数据来源或用户身份 | 口令、证书、质询应答、Kerberos | 认证解决“是谁” |
| 不可抵赖性 | 行为发生后不能否认 | 数字签名、审计日志、时间戳 | MAC 通常不提供对第三方的不可抵赖 |

答题模板：

```text
某攻击破坏了什么属性？
先判断攻击后果：泄露信息 -> 保密性；篡改内容 -> 完整性；服务不可访问 -> 可用性；冒充身份 -> 可认证性；事后否认签署/发送 -> 不可抵赖性。
```

### 1.2 主动攻击与被动攻击

被动攻击不改变系统状态，典型是窃听、流量分析。主动攻击会改变数据、身份或服务状态，典型是伪装、重放、篡改、ARP 欺骗、DNS 欺骗、拒绝服务。

常见考点：

- 嗅探通常是被动攻击。
- ARP/DNS 欺骗是主动攻击。
- DoS/DDoS 主要破坏可用性。
- 重放攻击是身份认证协议的核心威胁之一。

## 2. 第二章 常用密码算法

### 2.1 古典密码：会分类、会举例

古典密码常用于客观题，重点不是高强度计算，而是知道它们属于替代/置换、多表/单表。

凯撒密码是单表替代密码：

```text
C = (P + k) mod 26    # P 是明文字母编号，k 是位移密钥，C 是密文字母编号；每个字母统一后移 k 位
P = (C - k) mod 26    # 解密时按相同 k 反向移动；mod 26 表示超过 Z 后从 A 重新开始
```

例：`k=3`，`HELLO` 加密为 `KHOOR`。`H` 后移 3 位为 `K`，`E` 后移 3 位为 `H`。

单表替代密码使用一个固定替代表，弱点是字母频率仍会暴露。维吉尼亚密码是多表替代密码，通过重复密钥控制每个位置的位移，但如果密钥周期短，仍可能受到统计分析。

Kerckhoffs 假设：密码系统安全性不应依赖算法保密，而应依赖密钥保密。考试中看到“算法公开后仍应安全”，对应这个思想。

### 2.2 对称密码算法代表

对称密码指加密和解密使用相同密钥，或两个密钥可以容易互推。

```text
C = E_K(P)    # P 是明文，K 是共享密钥，E 是加密算法；发送方用 K 把 P 加密成密文 C
P = D_K(C)    # C 是密文，K 仍是同一个共享密钥，D 是解密算法；接收方用 K 恢复明文 P
```

代表算法：

- DES：传统分组密码，64 bit 分组，有效密钥 56 bit，16 轮 Feistel 结构。
- 3DES：多次 DES 组合，安全性高于 DES，但速度慢。
- AES：现代主流分组密码，分组长度 128 bit，密钥长度 128/192/256 bit。
- RC4：流密码，曾用于 WEP/TLS，现代协议中不推荐。
- SM4：我国商用密码中的对称分组密码，分组长度 128 bit，密钥长度 128 bit。

优点：速度快，适合大量数据加密。

缺点：密钥分发困难。`N` 个用户两两安全通信需要 `N(N-1)/2` 个共享密钥，规模大时难以管理。因此实际系统常用“公钥密码分发会话密钥 + 对称密码加密数据”的混合方案。

### 2.3 DES 输入输出、Feistel 结构和 S 盒

DES 客观题/计算题必记：

- 输入：64 bit 明文分组。
- 密钥：通常写 64 bit，其中 8 bit 为奇偶校验位，有效密钥 56 bit。
- 输出：64 bit 密文分组。
- 结构：初始置换、16 轮 Feistel、左右交换、逆初始置换。
- 每轮 F 函数：扩展置换、与子密钥异或、S 盒代换、P 盒置换。

Feistel 一轮：

```text
L_i = R_{i-1}                              # L_i 是第 i 轮左半块；直接取上一轮右半块 R_{i-1}
R_i = L_{i-1} xor F(R_{i-1}, K_i)          # R_i 是第 i 轮右半块；上一轮左半块与轮函数输出异或得到
```

`K_i` 是第 i 轮子密钥。Feistel 结构的关键是异或可逆，所以解密时轮函数不必可逆，只需把子密钥顺序反过来。

DES S 盒计算模板：

```text
输入 6 bit：b1 b2 b3 b4 b5 b6
行号 = b1b6          # 首位和末位组成 2 bit 行号
列号 = b2b3b4b5      # 中间 4 bit 组成列号
输出 = S[行号][列号] # 到题目给出的 S 盒表查值，再转成 4 bit 二进制
```

课堂例题形式：

```text
输入 101001
行号 = 11_2 = 3      # 首尾两位是 1 和 1，二进制 11 等于十进制 3
列号 = 0100_2 = 4    # 中间四位是 0100，等于十进制 4
若使用 DES 的 S1 盒，S1[3][4] = 4，输出 0100
```

考试若给出不同 S 盒表，应按题目表格查，不要死记某一个表值。

### 2.4 分组密码工作模式

ECB：

```text
C_i = E_K(P_i)    # P_i 是第 i 个明文块，K 是密钥；每块独立加密成 C_i
```

相同明文块会得到相同密文块，容易暴露图像、表格、重复数据结构。

CBC：

```text
C_1 = E_K(P_1 xor IV)       # IV 是初始化向量；第一块明文先与 IV 异或再加密
C_i = E_K(P_i xor C_{i-1})  # 第 i 块明文先与上一块密文异或，再加密得到 C_i
P_i = D_K(C_i) xor C_{i-1}  # 解密时先解 C_i，再与上一块密文异或恢复 P_i
```

`IV` 的作用是让相同明文在不同加密中得到不同密文，避免第一块泄露重复结构。

CTR：

```text
KS_i = E_K(Nonce || Counter_i)    # Nonce 是一次性值，Counter_i 是计数器；二者加密生成密钥流片段
C_i = P_i xor KS_i                # 明文 P_i 与密钥流 KS_i 异或得到密文 C_i
P_i = C_i xor KS_i                # 接收方用同一密钥流再异或一次恢复明文
```

CTR 的禁忌是重复使用同一 `Nonce || Counter`，否则密钥流重复，两个密文异或会暴露两个明文之间的关系。

### 2.5 非对称密码、加密与签名

非对称密码使用公钥和私钥。公钥公开，私钥保密。

公钥加密提供保密性：

```text
A -> B: C = E_{PU_B}(M)       # PU_B 是 B 的公钥；A 用 B 的公钥加密消息 M，得到密文 C
B: M = D_{PR_B}(C)            # PR_B 是 B 的私钥；只有 B 能用私钥从 C 解出 M
```

这不能证明消息来自 A，因为任何人都能用 B 的公钥加密。

数字签名提供来源认证、完整性和不可抵赖：

```text
h = H(M)                      # M 是消息，H 是哈希函数；先把消息压缩成固定长度摘要 h
s = Sign_{PR_A}(h)            # PR_A 是 A 的私钥；A 用私钥对摘要 h 签名得到 s
Verify_{PU_A}(H(M), s)        # PU_A 是 A 的公钥；B 用它验证签名 s 是否对应当前消息摘要
```

签名不提供保密性，因为消息通常仍可被看到。需要保密时应再加密消息。

Diffie-Hellman 密钥交换：

```text
公开参数：p, g                       # p 是大素数，g 是生成元；二者公开
Alice: A = g^a mod p                 # a 是 Alice 私密随机数；A 是可公开发送的 DH 公共值
Bob:   B = g^b mod p                 # b 是 Bob 私密随机数；B 是可公开发送的 DH 公共值
Alice: K = B^a mod p                 # Alice 用 Bob 的公开值 B 和自己的私钥 a 算共享密钥
Bob:   K = A^b mod p                 # Bob 用 Alice 的公开值 A 和自己的私钥 b 算同一个共享密钥
```

基本 DH 不认证身份，容易被中间人攻击。实际 TLS/IPsec 会把 DH 与证书、签名或预共享密钥结合。

### 2.6 RSA 计算题模板

RSA 密钥生成：

```text
n = p * q                    # p 和 q 是两个素数；n 是 RSA 模数，会出现在公钥和私钥中
phi(n) = (p - 1)(q - 1)       # phi(n) 是欧拉函数值，用于求公钥指数和私钥指数的逆元关系
gcd(e, phi(n)) = 1            # e 是公钥指数，必须与 phi(n) 互素
e * d ≡ 1 mod phi(n)          # d 是私钥指数，是 e 在模 phi(n) 下的乘法逆元
公钥 = (e, n)                 # 公钥可公开，用于加密或验证签名
私钥 = (d, n)                 # 私钥必须保密，用于解密或生成签名
```

加密与解密：

```text
C = M^e mod n    # M 是明文数字，e/n 是接收方公钥；发送方用公钥加密得到密文 C
M = C^d mod n    # C 是密文，d/n 是接收方私钥；接收方用私钥解密恢复 M
```

签名与验签可以理解为方向相反：

```text
s = H(M)^d mod n        # 发送者用自己的私钥指数 d 对消息摘要签名
H(M) = s^e mod n        # 验证者用发送者公钥指数 e 验证签名是否对应摘要
```

课堂例题 1：由 `p, q, e` 求私钥 `d`

```text
已知 p=5, q=11, e=3
n = p*q = 55                         # 模数 n 是 5 和 11 的乘积
phi(n) = (5-1)(11-1) = 40             # 欧拉函数值为 40
求 d：3*d ≡ 1 mod 40                  # d 要让 3*d 除以 40 余 1
d = 27                                # 3*27=81，81=2*40+1，所以 d=27
公钥 = (3,55)，私钥 = (27,55)
```

用小明文验证：

```text
M = 2
C = 2^3 mod 55 = 8                    # 用公钥加密，得到密文 8
M = 8^27 mod 55 = 2                   # 用私钥解密，可恢复 2；考试可用反复平方简化计算
```

课堂例题 2：若题目说“由私钥算公钥”

实际就是求 `e`，让 `e*d ≡ 1 mod phi(n)`。

```text
已知 p=5, q=11, d=27
phi(n)=40                             # 仍先计算欧拉函数
27*e ≡ 1 mod 40                       # e 是 d 在模 40 下的逆元
e = 3                                 # 27*3=81，81=2*40+1，所以公钥指数 e=3
n = 55                                # 公钥还需要模数 n
公钥 = (3,55)
```

易错点：

- RSA 明文数字必须满足 `M < n`。
- RSA 安全性主要与大整数分解困难有关。
- RSA 运算慢，实际常用来加密会话密钥或签名摘要，不直接加密大文件。

### 2.7 数字信封：混合加密思想

数字信封兼顾对称加密效率和公钥加密的密钥分发能力。

```text
K_s = 随机生成的一次性会话密钥          # K_s 用于本次通信，适合加密大量数据
C = E_{K_s}(M)                          # M 是正文；用对称密钥 K_s 加密得到正文密文 C
Envelope = E_{PU_B}(K_s)                # PU_B 是接收方 B 的公钥；用它加密 K_s，形成数字信封
B: K_s = D_{PR_B}(Envelope)             # PR_B 是 B 的私钥；只有 B 能打开数字信封取出 K_s
B: M = D_{K_s}(C)                       # B 再用 K_s 解密正文密文 C，恢复消息 M
```

关键词：正文用对称密钥加密，速度快；会话密钥用接收方公钥加密，分发安全。

## 3. 第三章 报文鉴别与哈希函数

### 3.1 哈希函数作用

哈希函数把任意长度输入映射为固定长度摘要。

```text
h = H(M)    # M 是任意长度消息，H 是哈希函数；h 是固定长度摘要，可看作消息指纹
```

安全哈希函数应具备：

- 单向性：由 `h` 反推出 `M` 在计算上不可行。
- 弱抗碰撞：给定 `M`，难以找另一个 `M'` 使 `H(M)=H(M')`。
- 强抗碰撞：难以找到任意两个不同消息具有相同摘要。
- 雪崩效应：输入微小变化会导致摘要明显变化。

常见应用：

- 文件完整性校验。
- 数字签名前先算摘要。
- 口令摘要存储。
- 软件发布校验。
- 区块链交易和区块头摘要。

### 3.2 MAC、哈希、数字签名对比

| 技术 | 是否用密钥 | 主要作用 | 是否支持第三方验证 | 易错点 |
|---|---|---|---|---|
| 哈希 | 不用密钥 | 完整性指纹 | 不能单独证明来源 | 哈希值若也被攻击者替换，校验会失效 |
| MAC | 使用共享密钥 | 完整性、共享密钥持有者认证 | 通常不支持 | 双方都知道密钥，难以证明到底谁生成 |
| 数字签名 | 私钥签名、公钥验证 | 完整性、身份认证、不可抵赖 | 支持 | 依赖公钥可信绑定，即 PKI/证书 |

HMAC 思想：

```text
tag = HMAC_K(M)    # K 是共享密钥，M 是消息；tag 是消息认证码，用于验证完整性和共享密钥持有者身份
```

### 3.3 生日攻击

若哈希输出长度为 `n` bit，输出空间为 `2^n`，寻找任意碰撞的复杂度约为：

```text
2^(n/2)    # n 是摘要位数；生日悖论使任意碰撞所需尝试次数约为输出空间平方根
```

注意：生日攻击针对“任意两个消息碰撞”，不是指定某个消息的前像。

### 3.4 数字签名与哈希结合

直接对大文件签名效率低，所以通常先哈希再签名。

```text
h = H(M)                         # M 是原始消息；先计算摘要 h，压缩消息长度
s = Sign_{PR_A}(h)               # PR_A 是 A 的私钥；A 对摘要 h 签名得到 s
A -> B: M, s, Cert_A             # Cert_A 是 A 的证书；B 用它获取可信的 A 公钥
B: h1 = H(M)                     # B 对收到的消息重新计算摘要
B: Verify_{PU_A}(h1, s)          # PU_A 是证书中的 A 公钥；验证签名和消息是否匹配
```

若 `M` 被改，`h1` 会变；若签名不是 A 私钥生成，验签失败。

### 3.5 综合分析题：下载文件哈希值与公告不一致

题目问“下载文件和公告里的哈希值不一样是为什么”，答题应从技术和安全后果两方面展开。

可能原因：

- 文件下载过程中损坏，网络传输或磁盘保存出错。
- 下载到的版本和公告对应版本不同。
- 文件被篡改，可能被植入恶意代码。
- 公告中的哈希值写错或没有及时更新。
- 下载站点被劫持，攻击者替换了文件。
- 更严重时，攻击者同时替换文件和网页上的哈希值，所以应从可信渠道或数字签名验证。

标准答题：

```text
H(file_downloaded) != H(file_announced)
```

这说明下载文件的内容与公告所对应的内容不一致。哈希函数具有雪崩效应，哪怕只改 1 bit，摘要也会明显变化。因此不能继续信任该文件，应重新从可信来源下载，并优先验证发布者数字签名或 HTTPS 证书链。

易错点：哈希不提供保密性，也不单独提供身份认证。只有当哈希值来自可信渠道时，文件校验才有意义。

## 4. 第四章 PKI 及其应用

### 4.1 PKI 解决什么问题

公钥密码允许公钥公开，但还要解决“这个公钥到底属于谁”。PKI 的核心就是把身份和公钥可信绑定。

数字证书本质：

```text
Cert_A = {A 的身份信息, PU_A, 有效期, 序列号, 签发者, CA 签名}
```

`PU_A` 是 A 的公钥。CA 的签名表示 CA 为“这个公钥属于这个主体”背书。

### 4.2 PKI 常见组成部分

客观题多选重点：

- CA：证书认证机构，负责签发证书和签发 CRL。
- RA：注册机构，负责证书申请者身份审核、注册受理。
- 证书发布系统/证书库：发布证书，常见方式包括 LDAP、HTTP。
- 证书状态发布机制：发布 CRL，或提供 OCSP 在线状态查询。
- PKI 客户端软件：如浏览器，负责证书下载、验证、状态检查。
- PKI 策略和 CPS：规定证书签发、撤销、密钥管理等操作规则。
- 支撑软硬件系统：服务器、数据库、安全模块、网络环境等。

简化记忆：`CA 签发，RA 审核，目录发布，CRL/OCSP 查状态，客户端验证，CPS 定规则`。

### 4.3 X.509 数字证书内容

证书常见字段：

- 版本号。
- 序列号。
- 签名算法标识符。
- 颁发者 Issuer。
- 有效期。
- 主体 Subject。
- 主体公钥信息。
- 扩展字段，如密钥用途、CRL/OCSP 地址、证书策略。
- CA 对证书主体内容的数字签名。

易错点：数字证书中不应包含主体私钥明文。证书公开发布，私钥只能由主体自己保存。

### 4.4 证书链验证

浏览器验证网站证书时，一般检查：

1. 证书链是否能追溯到本地信任的根 CA。
2. 每一级证书签名是否正确。
3. 证书是否在有效期内。
4. 域名是否匹配证书中的 Subject Alternative Name。
5. 证书用途是否允许服务器认证。
6. 证书是否被撤销，即 CRL/OCSP 状态。

证书链例子：

```text
Root CA -> Intermediate CA -> www.example.com
```

根 CA 是操作系统/浏览器预置信任点；中间 CA 由根 CA 签发；网站证书由中间 CA 签发。

### 4.5 CRL 是什么，有什么用

CRL 是 `Certificate Revocation List`，证书撤销列表。

```text
CRL = Sign_{PR_CA}(已撤销证书序列号列表, 本次更新时间, 下次更新时间)
```

`PR_CA` 是 CA 私钥。CA 对撤销列表签名，验证方用 CA 公钥确认 CRL 没被篡改。

CRL 的作用：

- 记录尚未过期但已经被撤销的证书。
- 典型撤销原因：私钥泄露、身份信息变化、证书误签发、权限取消。
- 验证方通过证书序列号查询证书是否在 CRL 中。

CRL 撤销过程答题版：

```text
1. 用户或机构向 CA 申请撤销证书
# 例如私钥泄露、人员离职、证书信息错误或机构停用该证书

2. CA 确认证书需要撤销
# CA 要确认撤销请求合法，避免攻击者恶意撤销他人证书

3. CA 将证书序列号、撤销时间、撤销原因加入 CRL
# CRL 主要按证书序列号识别被撤销证书，而不是按用户名简单匹配

4. CA 用自己的私钥对 CRL 签名
# 验证方可以用 CA 公钥确认 CRL 是 CA 发布且未被篡改

5. CA 发布 CRL
# CRL 可通过 HTTP、LDAP 等方式发布，证书扩展字段中常包含 CRL 分发点 URL

6. 验证方下载或查询 CRL，验证签名并检查目标证书序列号
# 若目标证书序列号在 CRL 中，即使证书未过期，也应视为不可信
```

CRL 局限：

- 定期发布，存在时间滞后。
- 大型 CA 的 CRL 可能很大。
- 用户或浏览器不检查 CRL 时，撤销信息不能发挥作用。

### 4.6 数字证书状态检查作用

证书“未过期”不等于“仍可信”。状态检查用于确认证书是否已被 CA 撤销。

两种方式：

- CRL：下载撤销列表，本地查证书序列号。
- OCSP：在线向状态服务器查询某张证书当前状态。

答题模板：

```text
证书状态检查用于发现“未到期但已失效”的证书。例如服务器私钥泄露后，证书有效期还没结束，但 CA 会撤销证书。客户端检查 CRL 或 OCSP，若发现证书已撤销，就应拒绝信任该证书。
```

## 5. 第五章 身份认证

### 5.1 身份认证基本方法

身份认证是动态协议，重点是证明“当前交互的一方确实是它声称的身份”。最大威胁是重放攻击，所以需要鲜活性。

认证因素：

- 所知：口令、PIN。
- 所有：U 盾、智能卡、动态令牌。
- 所是：指纹、人脸、虹膜、声纹。

双因素认证必须来自不同类别。例如“口令 + 手机动态码”是两类；“两个不同口令”仍主要是同一类。

### 5.2 口令认证

不安全做法：服务器保存明文口令。

较安全做法：保存加盐哈希。

```text
注册：保存 salt, H(salt || password)              # salt 是随机盐值；password 是用户口令；系统保存哈希而不是明文
登录：计算 H(salt || password') 并比较保存值       # password' 是本次输入；结果一致说明输入口令与注册口令一致
```

`salt` 不是秘密，它的作用是让相同口令在不同用户处得到不同哈希，增加彩虹表和预计算字典攻击成本。

易错点：

- 哈希口令不等于绝对安全，弱口令仍会被离线字典攻击。
- 传输登录口令时仍需要 TLS 等安全通道。
- 口令认证本身容易被钓鱼和重放，需要动态因素或质询应答增强。

### 5.3 动态口令

动态口令是随时间、计数器或事件变化的一次性认证值。它降低旧口令被重放的风险。

Lamport 哈希链例子：

```text
用户选择秘密 w                         # w 是种子，只由用户掌握
计算 w, h(w), h(h(w)), ..., h^t(w)      # h^t(w) 表示对 w 连续哈希 t 次
服务器初始保存 h^t(w)                  # 服务器保存链尾，不保存种子 w
第一次登录提交 h^{t-1}(w)              # 用户提交链尾前一个值，作为一次性口令
服务器验证 h(h^{t-1}(w)) = h^t(w)       # 服务器哈希一次后若等于保存值，说明口令正确
服务器更新保存值为 h^{t-1}(w)           # 旧值作废，下次必须提交更前一个链值
```

动态口令的作用：

- 每次认证值不同。
- 截获旧口令也不能直接用于下一次。
- 与时间/计数器/哈希链绑定，降低重放成功概率。

新 Word 中的动态口令生成和验证机制可按下面写：

```text
OTP = HMAC_Seed(T 或 Counter)
# Seed 是客户端和服务器预先共享的秘密种子；T 是当前时间片，Counter 是计数器；HMAC/Hash 生成一次性口令

客户端提交：用户名, OTP
# 用户名用于让服务器找到该用户对应的 Seed；OTP 是本次动态口令

服务器计算：OTP' = HMAC_Seed(T 或 Counter)
# 服务器用同一个 Seed、同一算法和当前时间片/计数器独立计算本地口令 OTP'

比较 OTP 与 OTP'
# 两者一致则认证通过；不一致说明口令错误、过期或计数器/时间不同步
```

其中时间型动态口令要求客户端和服务器时间大致同步，计数器型动态口令要求双方计数器同步。实际系统常允许很小的时间窗口，避免因为几秒钟误差导致合法用户失败。

局限：如果攻击者实时钓鱼并立即转发，动态口令仍可能被利用。因此真实系统会结合域名证书、交易确认、风控等机制。

### 5.4 质询应答与随机数作用

质询应答的核心是验证方给出本轮随机挑战，宣称方用秘密处理挑战。

基本形式：

```text
B -> A: rB                         # B 是验证方；rB 是 B 生成的一次性随机数，用于保证本轮认证新鲜
A -> B: response = f(secret, rB)    # A 用自己的秘密 secret 和 rB 计算响应；响应必须随 rB 改变
B: 验证 response                    # B 检查 response 是否正确，从而确认 A 知道 secret
```

随机数 `rB` 的作用：

- 让每次认证输入不同。
- 绑定本次会话，防止旧响应重放。
- 可参与后续密钥派生，使会话密钥每次不同。

对称加密单向认证：

```text
B -> A: rB             # B 发送随机挑战 rB；攻击者无法提前知道新挑战
A -> B: E_K(rB, B)     # K 是 A/B 共享密钥；A 加密 rB 和接收方身份 B，证明自己知道 K 并回应的是 B
```

加入 `B` 是为了绑定接收方身份，减少反射攻击风险。

MAC 双向认证：

```text
B -> A: rB                          # B 发出随机数 rB，要求 A 证明身份
A -> B: rA, MAC_K(rA, rB, B)         # rA 是 A 的随机数；MAC 绑定双方随机数和 B 的身份，证明 A 知道 K
B -> A: MAC_K(rA, rB, A)             # B 返回 MAC，绑定同一组随机数和 A 的身份，证明 B 也知道 K
```

签名认证：

```text
B -> A: rB                                      # B 发送一次性随机挑战 rB，防止 A 重放旧签名
A -> B: Cert_A, rA, B, Sign_{PR_A}(rA, rB, B)   # Cert_A 是 A 的证书；PR_A 是 A 私钥；签名绑定双方随机数和接收方 B
B: Verify_{PU_A}(Sign, rA, rB, B)               # PU_A 来自证书；B 验证签名，确认 A 持有对应私钥
```

### 5.5 Kerberos：三类角色、票据和交互关系

老师提到的“不同角色、认证过程、互动关系”要按 Kerberos 三方/四实体来记。

角色：

- C：Client，客户端/用户。
- AS：Authentication Server，认证服务器，负责初始登录认证。
- TGS：Ticket Granting Server，票据许可服务器，负责给不同应用服务发票。
- V：Application Server，应用服务器，提供具体服务。

三项作用：

- AS 做初始身份认证：确认用户是谁。
- TGS 做授权换票：用户拿 TGT 换访问某个服务器的票。
- V 做服务访问验证：服务器检查服务票据和认证子，确认客户端当前有效。

关键密钥：

- `K_C`：由用户口令导出的客户端密钥，C 和 AS 知道。
- `K_TGS`：AS 与 TGS 共享密钥。
- `K_V`：TGS 与应用服务器 V 共享密钥。
- `K_C,TGS`：AS 为 C 和 TGS 生成的会话密钥。
- `K_C,V`：TGS 为 C 和 V 生成的会话密钥。

Kerberos V4 简化 6 步：

```text
1. C -> AS: ID_C, ID_TGS, TS1
# ID_C 是客户端身份，ID_TGS 是请求访问的票据服务器，TS1 是时间戳；C 向 AS 申请登录后的票据许可票

2. AS -> C: E_{K_C}(K_C,TGS, ID_TGS, TS2, Lifetime, Ticket_TGS)
# K_C 是由用户口令导出的密钥；AS 用 K_C 加密返回 C/TGS 会话密钥 K_C,TGS、时间戳、有效期和票据

Ticket_TGS = E_{K_TGS}(K_C,TGS, ID_C, AD_C, ID_TGS, TS2, Lifetime)
# Ticket_TGS 是给 TGS 的票据；K_TGS 是 AS/TGS 共享密钥，所以客户端 C 只能转交，不能解开或篡改

3. C -> TGS: ID_V, Ticket_TGS, Authenticator_C
# ID_V 是要访问的应用服务器；C 把 Ticket_TGS 交给 TGS，并附上认证子证明自己是当前会话参与者

Authenticator_C = E_{K_C,TGS}(ID_C, AD_C, TS3)
# K_C,TGS 是 C/TGS 会话密钥；认证子包含身份、客户端地址和时间戳，防止重放

4. TGS -> C: E_{K_C,TGS}(K_C,V, ID_V, TS4, Ticket_V)
# TGS 验证 Ticket_TGS 和认证子后，返回 C/V 会话密钥 K_C,V，以及访问 V 的服务票据

Ticket_V = E_{K_V}(K_C,V, ID_C, AD_C, ID_V, TS4, Lifetime)
# Ticket_V 是给应用服务器 V 的票据；K_V 是 TGS/V 共享密钥，只有 V 能解开

5. C -> V: Ticket_V, Authenticator'_C
# C 访问 V 时提交服务票据和新的认证子；V 用 Ticket_V 得到 K_C,V，再验证认证子

Authenticator'_C = E_{K_C,V}(ID_C, AD_C, TS5)
# K_C,V 是 C/V 会话密钥；TS5 证明请求是新鲜的，不是旧请求重放

6. V -> C: E_{K_C,V}(TS5 + 1)
# V 用同一会话密钥返回 TS5+1，证明 V 也知道 K_C,V，从而实现双向认证
```

Kerberos 为什么能单点登录：用户只在登录时向 AS 认证一次，获得 `Ticket_TGS`。之后访问不同服务器时，用 `Ticket_TGS` 向 TGS 换不同 `Ticket_V`，不需要把用户口令交给每个应用服务器。

易错点：

- Kerberos 主要基于对称密钥，不是 PKI 证书体系。
- Kerberos 依赖时间戳，要求主机时钟同步。
- Ticket 是给服务器看的，客户端通常只能转发，不能解密修改。

## 6. 第六章 Web 与电子商务安全

### 6.1 SSL/TLS 提供的安全能力

客观题重点：SSL/TLS 为通信双方提供安全通道，核心能力包括：

- 服务器身份认证。
- 可选的客户端身份认证。
- 会话密钥协商。
- 应用数据保密性。
- 应用数据完整性。
- 防止握手过程被篡改。

TLS 协议组成：

- 握手协议：协商版本、算法、随机数、认证服务器、交换密钥。
- 记录协议：分段、压缩可选、加密、完整性保护、封装应用数据。
- ChangeCipherSpec：通知后续记录切换到协商出的密码套件和密钥。
- Alert：传递错误和关闭通知。

易错点：TLS 保护传输过程，不等于网站程序本身没有漏洞，也不等于用户终端没有木马。

### 6.2 TLS/SSL 握手协议流程（图 9.6 类题）

第一阶段：Hello 协商。

```text
Client -> Server: ClientHello
# ClientHello 携带客户端支持的 TLS 版本、客户端随机数 ClientRandom、会话 ID、密码套件列表和扩展

Server -> Client: ServerHello
# ServerHello 选择 TLS 版本、密码套件、压缩方法，返回服务器随机数 ServerRandom
```

作用：双方协商使用什么算法，并交换两个随机数。随机数会参与后续密钥派生，保证每次连接密钥不同。

第二阶段：服务器认证和密钥交换参数。

```text
Server -> Client: Certificate
# Certificate 是服务器证书；客户端用它获得服务器公钥，并验证证书链、域名、有效期、用途和撤销状态

Server -> Client: ServerKeyExchange
# 某些密钥交换方式需要该消息；它携带服务器密钥交换参数，服务器通常对参数签名以防篡改

Server -> Client: CertificateRequest 可选
# 服务器若需要客户端证书认证，会用该消息要求客户端提交证书

Server -> Client: ServerHelloDone
# 表示服务器这一轮握手消息发送完毕，轮到客户端继续
```

作用：客户端确认服务器身份，并获得协商密钥所需参数。

第三阶段：客户端密钥交换和可选客户端认证。

```text
Client -> Server: ClientKeyExchange
# 携带客户端密钥交换材料；RSA 方式下通常是用服务器公钥加密的 premaster_secret，DH 方式下是客户端 DH 公共值

Client -> Server: Certificate 可选
# 如果服务器请求客户端认证，客户端发送自己的证书

Client -> Server: CertificateVerify 可选
# 客户端用自己的私钥对握手消息摘要签名，证明自己持有客户端证书对应私钥
```

传统 RSA 密钥交换中：

```text
Client -> Server: E_{PU_Server}(premaster_secret)
# PU_Server 是服务器证书里的公钥；客户端用它加密预主密钥 premaster_secret

Server: premaster_secret = D_{PR_Server}(...)
# PR_Server 是服务器私钥；只有真正服务器能解出 premaster_secret

master_secret = PRF(premaster_secret, ClientRandom, ServerRandom)
# PRF 是密钥派生函数；双方用预主密钥和两个随机数派生主密钥
```

作用：双方得到后续记录协议使用的对称密钥材料。

第四阶段：切换密码并验证握手完整性。

```text
Client -> Server: ChangeCipherSpec
# 客户端声明后续发送的记录将使用协商出的密码套件和密钥保护

Client -> Server: Finished
# Finished 用新密钥保护，校验此前所有握手消息是否一致、是否被篡改

Server -> Client: ChangeCipherSpec
# 服务器也声明后续记录切换到新密钥

Server -> Client: Finished
# 服务器同样发送 Finished，证明自己看到的握手过程与客户端一致
```

Finished 是握手安全闭环：如果中间人改过前面的握手消息，Finished 校验会失败。

### 6.3 SET 协议：双重数字签名与数字信封

SET 面向在线信用卡交易，参与者包括持卡人、商家、支付网关、发卡行、收单行和 CA。它和 TLS 的区别是：TLS 主要保护两方传输通道，SET 保护多方交易关系，并强调隐私隔离。

SET 的三个核心安全目标：

- 安全交易通道：保密性和完整性。
- 信任：基于 X.509 证书和公钥密码完成身份认证。
- 隐私保护：不同参与方只能看到自己应该知道的信息。

SET 交易基本步骤：

1. 持卡人开通账号。
2. 持卡人获得证书。
3. 商家获得自己的证书。
4. 持卡人下订单。
5. 持卡人验证商家身份。
6. 持卡人将订单信息和支付信息发送给商家。
7. 商家请求支付授权。
8. 商家确认订单。
9. 商家提供货物或服务。
10. 商家请求支付。

第 6 步是重点，因为它既要让商家知道订单，又不能让商家看到完整信用卡支付信息；也要让银行看到支付信息，又不让银行看到完整订单明文。

符号：

- `PI`：Payment Information，支付信息，如卡号、有效期、持卡人身份。
- `OI`：Order Information，订单信息，如商品、数量、价格。
- `PIMD`：支付信息摘要。
- `OIMD`：订单信息摘要。
- `POMD`：支付和订单摘要组合后的摘要。
- `DS`：Dual Signature，双重数字签名。
- `K_s`：一次性对称会话密钥。
- `KR_C`：持卡人私钥。
- `KU_C`：持卡人公钥。
- `KU_Bank`：银行/支付网关公钥。
- `KR_Bank`：银行/支付网关私钥。

双重数字签名生成：

```text
PIMD = H(PI)
# PI 是支付信息；PIMD 是支付信息摘要，商家可以拿到摘要但看不到支付明文

OIMD = H(OI)
# OI 是订单信息；OIMD 是订单信息摘要，银行可以拿到摘要但看不到订单明文

POMD = H(PIMD || OIMD)
# || 表示连接；POMD 把支付摘要和订单摘要绑定，防止支付信息和订单信息被拆开替换

DS = E_{KR_C}(POMD)
# KR_C 是持卡人私钥；用私钥对 POMD 签名得到 DS，证明持卡人认可这组订单和支付信息
```

支付信息数字信封：

```text
C_pay = E_{K_s}(PI, OIMD, DS)
# K_s 是一次性对称密钥；用它加密支付信息 PI、订单摘要 OIMD 和双重签名 DS，形成银行可解的支付密文

Envelope = E_{KU_Bank}(K_s)
# KU_Bank 是银行/支付网关公钥；用它加密 K_s，形成数字信封，只有银行能用私钥打开
```

持卡人发给商家的 Purchase Request 可理解为两部分：

```text
商家可读部分：OI, PIMD, DS, Cert_C
# 商家需要订单明文 OI，支付摘要 PIMD，持卡人签名 DS 和持卡人证书 Cert_C，用于验证订单和持卡人身份

银行可读部分：C_pay, Envelope
# C_pay 和 Envelope 商家打不开，只能原封不动转交给支付网关/银行
```

商家验证：

```text
OIMD' = H(OI)
# 商家对自己看到的订单明文 OI 重新计算摘要 OIMD'

POMD' = H(PIMD || OIMD')
# 商家把收到的支付摘要 PIMD 和自己计算的订单摘要 OIMD' 绑定计算 POMD'

POMD_from_DS = D_{KU_C}(DS)
# KU_C 来自持卡人证书；商家用持卡人公钥验证签名，得到持卡人签过的 POMD

比较 POMD' 与 POMD_from_DS
# 两者一致，说明订单信息未被改，且该订单和某个支付信息被持卡人绑定签名
```

银行/支付网关验证：

```text
K_s = D_{KR_Bank}(Envelope)
# KR_Bank 是银行私钥；银行打开数字信封，取出对称会话密钥 K_s

PI, OIMD, DS = D_{K_s}(C_pay)
# 银行用 K_s 解密支付密文，得到支付信息 PI、订单摘要 OIMD 和双重签名 DS

PIMD' = H(PI)
# 银行对支付明文 PI 计算支付摘要 PIMD'

POMD' = H(PIMD' || OIMD)
# 银行把支付摘要和收到的订单摘要绑定计算 POMD'

POMD_from_DS = D_{KU_C}(DS)
# 银行用持卡人公钥验证双重签名，得到持卡人签过的 POMD

比较 POMD' 与 POMD_from_DS
# 一致则说明支付信息和订单摘要确实被持卡人绑定，银行可继续支付授权
```

为什么能保护隐私：

- 商家只看到 `OI` 和 `PIMD`，看不到 `PI` 明文。
- 银行看到 `PI` 和 `OIMD`，看不到 `OI` 明文。
- 双方都能通过 `DS` 确认订单和支付属于同一笔交易。

易错点：SET 双重签名不是“双重加密”。核心是把订单摘要和支付摘要绑定，同时隔离商家和银行的可见信息。

## 7. 第七章 TCP/IP 安全隐患与网络攻击

### 7.1 DNS 欺骗：原理、证据和防护

DNS 正常作用是把域名解析成 IP 地址。DNS 欺骗让用户得到错误的域名-IP 映射，从而访问攻击者指定主机。

典型攻击路径：

```text
用户 -> DNS: 查询 www.bank.com
# 用户希望得到真实银行网站的 IP

攻击者 -> 用户: www.bank.com = IP_Attacker
# 攻击者伪造或抢先返回 DNS 响应，把域名指向攻击者服务器

用户浏览器 -> IP_Attacker
# 用户看到域名可能没变，但实际连接到了攻击者控制的 IP
```

常见技术原理：

- DNS 缓存中毒：向递归 DNS 缓存写入错误记录。
- 伪造 DNS 响应：攻击者抢在真实 DNS 响应之前返回假结果。
- ARP 欺骗配合 DNS 欺骗：攻击者先成为局域网中间人，再篡改 DNS 响应。
- DHCP 欺骗：伪造 DHCP 服务器给客户端下发恶意 DNS 地址。
- hosts 文件篡改：本机静态解析被改写。
- 域名劫持：域名注册或解析管理权限被攻击者控制。

hosts 文件欺骗例子：

```text
IP_Attacker www.example.com
# IP_Attacker 是攻击者控制的 IP；操作系统若优先读取 hosts 文件，浏览器访问 www.example.com 时会直接连到攻击者地址
```

如果题目描述“别人访问网站都正常，只有某一台主机异常”，优先怀疑这台主机本地的 hosts 文件、DNS 缓存、DNS 配置或局域网内伪造响应，而不是首先怀疑目标网站服务器整体故障。

实验/综合分析题答题线索：

- 用户输入的域名正确，但解析到陌生 IP。
- 同一域名在可信 DNS 和本机 DNS 缓存中的结果不同。
- 抓包可见 DNS 响应源异常、事务 ID/端口/时序可疑。
- 若使用 HTTPS，浏览器可能出现证书域名不匹配或证书不可信警告。

验证方法：

```text
ping 域名
# 查看该域名当前解析到哪个 IP，并与正常主机或可信 DNS 的结果比较

nslookup 域名
# 直接查询 DNS 解析结果，观察返回 IP 和使用的 DNS 服务器是否异常

检查 hosts 文件
# 查看是否存在“攻击者 IP + 目标域名”的静态映射

清理 DNS 缓存后重新解析
# 若清理后恢复正常，说明本地缓存污染可能性较高

对比其他正常主机解析结果
# 若同一网络中只有一台异常，问题更可能在该主机本地配置或缓存
```

防护技术：

- DNSSEC：对 DNS 记录做数字签名，验证真实性和完整性。
- 随机化 DNS 事务 ID 和源端口，降低伪造响应命中概率。
- 限制递归 DNS 服务器只服务可信客户端，防止缓存污染。
- 使用可信 DNS 服务器，清理异常 DNS 缓存和 hosts 文件。
- 重要业务使用 HTTPS，并严格检查证书链、域名和撤销状态。
- 局域网启用静态 ARP、动态 ARP 检测、DHCP Snooping，减少中间人条件。
- 对 DNS 服务器做访问控制、日志审计和异常解析监测。

注意：DNSSEC 保护记录真实性和完整性，不负责加密查询内容。

### 7.2 拒绝服务攻击与防护

DoS/DDoS 目标是可用性。攻击者通过消耗带宽、CPU、内存、连接表或应用资源，使合法用户无法正常访问。

SYN Flood 原理：

```text
Client -> Server: SYN
# SYN 是 TCP 连接请求；服务器收到后通常要分配半连接资源

Server -> Client: SYN/ACK
# 服务器回复 SYN/ACK，并等待客户端 ACK 完成三次握手

攻击者不回复 ACK
# 半连接长期占用队列；大量伪造 SYN 会耗尽服务器半连接资源
```

SYN Flood 防护：

- 缩短 SYN Timeout。
- 增大半连接队列。
- SYN Cookie：收到 SYN 时暂不分配完整资源，把状态编码进序列号，等 ACK 正确返回再分配资源。
- 按源 IP 连接速率限制和异常阈值阻断。
- 防火墙、IPS、负载均衡、流量清洗。

Smurf 攻击：

```text
攻击者伪造源 IP = Victim
# 响应会发给受害者 Victim，而不是攻击者

攻击者 -> 广播地址: ICMP Echo Request
# 广播网络内多个主机会收到请求

多个主机 -> Victim: ICMP Echo Reply
# 大量响应汇聚到受害者，形成放大流量
```

Smurf 防护：

- 禁用定向广播。
- 路由器做源地址过滤，阻止伪造源 IP。
- 限制 ICMP 广播响应。
- 上游运营商流量清洗。

DDoS 综合防护：

- CDN/Anycast 分散流量。
- 黑洞路由或牵引清洗。
- 速率限制、连接数限制、验证码/挑战。
- 冗余部署和弹性扩容。
- 日志、监控、应急预案。

易错点：DDoS 不一定利用漏洞，很多请求形式上合法，所以仅靠修补漏洞不够。

### 7.3 IPsec：AH/ESP 与传输/隧道模式

IPsec 位于网络层，常用于 VPN、主机到主机、主机到网关或网关到网关的安全通信。

两个核心协议：

- AH：Authentication Header，提供数据源认证、完整性和抗重放，不提供加密保密性。
- ESP：Encapsulating Security Payload，可提供保密性，也可提供完整性、认证和抗重放，实际更常用。

两种模式：

```text
传输模式：保护 IP 载荷，原 IP 头保留
# 常用于主机到主机；原始源/目的 IP 地址仍在外层 IP 头中

隧道模式：把整个原 IP 包封装进新的 IP 包
# 常用于网关到网关 VPN；原始 IP 包被整体保护，外面再套一个新 IP 头
```

SA 与 IKE：

- SA：Security Association，安全关联，描述单向安全通信的算法、密钥、SPI、生命周期等参数。
- IKE：自动协商密钥和 SA。

客观题抓对比：`AH 重认证完整性，ESP 可加密；传输模式保护载荷，隧道模式封装整个原 IP 包`。

### 7.4 ARP、DHCP、路由欺骗与 DNS 欺骗关系

ARP 欺骗常是 DNS 欺骗的前置条件之一。

```text
攻击者 -> 目标主机: 网关 IP 对应 MAC_Attacker
# 目标误以为攻击者就是网关

攻击者 -> 网关: 目标 IP 对应 MAC_Attacker
# 网关误以为攻击者就是目标主机

目标主机 -> 攻击者 -> 网关
# 攻击者成为中间人，可嗅探、篡改或丢弃流量
```

DHCP 欺骗可下发恶意 DNS 或网关；路由欺骗可改变转发路径；IP 欺骗可用于隐藏攻击源或配合反射放大。

## 8. 第八章 防火墙技术

### 8.1 防火墙基本作用

防火墙是在不同网络区域之间实施访问控制的边界设备或系统。它定义内外网之间的必经点，根据规则允许或拒绝流量。

作用：

- 服务控制：允许或禁止访问某些服务。
- 方向控制：控制内到外、外到内流量。
- 用户控制：结合认证控制用户访问。
- 行为控制：限制上传、下载、命令等行为。
- 日志审计、报警、NAT、计费。
- 可作为 VPN/IPsec 平台。

局限：

- 不能保护绕过防火墙的连接。
- 不能完全防止内部攻击。
- 不能自动修复应用漏洞。
- 对加密隧道内部恶意内容可见性有限。
- 规则配置错误会导致防护失效。

### 8.2 包过滤与状态检测

包过滤依据源 IP、目的 IP、协议、端口、方向、TCP 标志位放行或拒绝。

规则例子：

```text
出方向：源 IP=内网，目的端口=80，TCP，允许
# 允许内网主机主动访问外部 Web 服务

入方向：源端口=80，ACK=1，目的 IP=内网，允许
# 允许外部 Web 服务器返回与已建立连接相关的响应包

默认：拒绝
# 未明确允许的流量全部丢弃，属于白名单策略
```

状态检测防火墙维护连接状态，比纯包过滤更能识别“是否属于已建立会话”。

应用层网关/代理代表内网客户端访问外部服务器，可以做更细的应用内容检查和用户认证，但性能开销大，每种应用通常需要专门代理。

### 8.3 防火墙部署方案

老师客观题重点：

1. 包过滤路由器方案  
   最简单，在内外网边界用包过滤路由器控制流量。优点是简单高效；缺点是层次少，防护能力有限。

2. 双宿主主机方案  
   堡垒主机有两块网卡，分别连接内外网，不直接路由转发，通常通过代理实现访问。优点是隔离性较强；缺点是堡垒主机压力大。

3. 屏蔽主机方案  
   包过滤路由器在外层，堡垒主机在内层。外部只能访问堡垒主机，内网受到一定保护。

4. 屏蔽子网方案  
   使用外部路由器、DMZ、内部路由器形成多层防护。对外服务器放在 DMZ，内网与外网进一步隔离。常见、较安全。

DMZ 作用：放置 Web、邮件、DNS 等对外服务，避免外部用户直接访问内网。

易错点：

- 防火墙不是万能安全设备。
- 默认拒绝比默认允许更保守。
- 低层过滤效率高但语义弱；高层代理控制细但开销大。

## 9. 第九章 区块链简介

### 9.1 区块链防篡改能力与哪些技术因素有关

区块链不是“绝对不可篡改”，而是通过多种机制提高篡改成本。

关键技术因素：

1. 哈希链接  
   每个区块头包含前一区块哈希。

```text
block_i.prev_hash = H(block_{i-1})
# block_i 保存前一区块的哈希；如果前一区块被改，H(block_{i-1}) 变化，链接立即对不上
```

2. Merkle 树  
   区块内交易组织成 Merkle 树，根哈希写入区块头。任何交易变化都会导致 Merkle Root 改变。

```text
Merkle_root = H(H(tx1) || H(tx2) || ...)
# tx 是交易；根哈希代表区块内交易集合，便于快速验证某交易是否属于区块
```

3. 工作量证明 PoW  
   矿工寻找 nonce，使区块头哈希满足难度目标。

```text
H(prev_hash, Merkle_root, timestamp, nonce, 其他头部字段) < Target
# nonce 是反复尝试的数；Target 越小，找到合格哈希越难，篡改后重算成本越高
```

4. 共识机制和最长链/累计工作量规则  
   节点接受累计工作量更大的链。攻击者若篡改历史区块，需要重算该块及后续区块，并超过诚实链。

5. 分布式存储和节点验证  
   多节点保存账本并独立验证，攻击者难以只修改一个地方就让全网接受。

6. 数字签名  
   交易由私钥签名，节点用公钥验证交易授权，防止他人伪造付款。

答题模板：

```text
区块链防篡改能力来自哈希链、Merkle 树、共识机制、工作量证明成本、分布式节点验证和交易数字签名。修改历史交易会改变交易哈希、Merkle 根和区块哈希，使后续区块链接失效；攻击者还必须重新完成后续区块的工作量并超过诚实链，因此篡改成本很高。
```

### 9.2 51% 攻击和确认数

若攻击者掌握超过诚实节点总和的算力，可能构造累计工作量更大的分支，改写近期交易。这说明区块链不是数学上绝对不可改，而是经济和算力上难以改。

交易等待多个确认，是为了降低所在区块被分叉回滚的概率。PPT 中常提到 6 个确认作为降低风险的经验值。

## 10. 综合题与简答题答题模板

### 10.1 简答：SET 数字信封如何实现

标准答题应包含四点：

1. 生成一次性对称密钥 `K_s`。
2. 用 `K_s` 加密支付信息等敏感内容。
3. 用银行/支付网关公钥加密 `K_s`，形成数字信封。
4. 银行用自己的私钥打开数字信封得到 `K_s`，再解密支付信息。

```text
C_pay = E_{K_s}(PI, OIMD, DS)
# 用对称密钥 K_s 加密支付信息 PI、订单摘要 OIMD 和双重签名 DS，效率高

Envelope = E_{KU_Bank}(K_s)
# 用银行公钥 KU_Bank 加密 K_s，只有银行能用私钥 KR_Bank 解开

K_s = D_{KR_Bank}(Envelope)
# 银行用私钥打开数字信封，取出会话密钥

PI, OIMD, DS = D_{K_s}(C_pay)
# 银行用会话密钥解密支付密文，得到支付信息并完成验证
```

### 10.2 简答：身份认证中随机数和动态口令的作用

随机数作用：

- 提供鲜活性。
- 防止旧认证报文重放。
- 绑定本次会话。
- 可参与会话密钥生成。

动态口令作用：

- 每次登录认证值不同。
- 旧口令截获后很快失效。
- 降低静态口令泄露风险。

答题时要强调：随机数和动态口令都不是为了“加密数据本身”，而是为了让认证凭证每次不同，抵抗重放攻击。

### 10.3 简答：CRL 是什么，有什么用

CRL 是 CA 签发的证书撤销列表，列出未过期但已被撤销的证书序列号。验证方检查 CRL，可以发现私钥泄露、身份变化、误签发等导致的失效证书。CRL 解决“证书未过期但不应再信任”的问题。

### 10.4 综合：DNS 欺骗分析题答法

按四段写：

1. 原理：DNS 把域名映射到 IP，欺骗攻击让用户得到错误 IP。
2. 过程：伪造响应、缓存中毒、ARP 中间人、DHCP 恶意 DNS 等。
3. 后果：访问钓鱼站、账号密码泄露、恶意软件下载、中间人攻击。
4. 防护：DNSSEC、可信 DNS、随机端口/事务 ID、限制递归、HTTPS 证书验证、ARP/DHCP 防护、日志监测。

### 10.5 综合：TLS 握手消息作用

按四阶段写：

1. Hello：协商版本、算法，交换随机数。
2. 服务器认证：服务器发证书和密钥交换参数，客户端验证证书。
3. 客户端密钥交换：客户端发送预主密钥或 DH 公共值，可选客户端证书认证。
4. ChangeCipherSpec/Finished：切换到协商密钥，验证握手完整性，之后保护应用数据。

### 10.6 计算：RSA 公钥/私钥指数

通用步骤：

```text
1. n = p*q
2. phi(n) = (p-1)(q-1)
3. 若已知 e，求 d：e*d ≡ 1 mod phi(n)
4. 若已知 d，求 e：d*e ≡ 1 mod phi(n)
5. 公钥 = (e,n)，私钥 = (d,n)
```

小数值题常用试算法：找一个整数 `k`，使 `e*d = k*phi(n)+1`。

### 10.7 计算：DES 输入输出和 S 盒

DES 基本输入输出：

```text
明文输入：64 bit
密钥输入：64 bit，其中 56 bit 有效，8 bit 奇偶校验
密文输出：64 bit
轮数：16 轮 Feistel
S 盒：6 bit 输入，4 bit 输出
```

S 盒步骤：

```text
行 = 第 1 位和第 6 位
列 = 中间 4 位
查表 = S[行][列]
输出 = 查表值转成 4 bit 二进制
```

### 10.9 新 Word 例题 1：RSA 完整计算与逐步解释

题目条件：

```text
p = 7
q = 13
e = 11
M = 9
```

要求：求 `n`、`phi(n)`、私钥指数 `d`、密文 `C`，并做解密验证。

第一步：计算模数 `n`。

```text
n = p * q = 7 * 13 = 91
# p 和 q 是 RSA 选取的两个素数；n 是公开模数，公钥和私钥都会用到 n
```

所以公钥中的模数为 `91`，公钥暂时可写为：

```text
(e, n) = (11, 91)
# e 是题目给出的公钥指数；n 是刚算出的模数
```

第二步：计算欧拉函数 `phi(n)`。

```text
phi(n) = (p - 1)(q - 1) = (7 - 1)(13 - 1) = 6 * 12 = 72
# p、q 是不同素数时，phi(n) 等于 (p-1)(q-1)；它用于求 e 和 d 的逆元关系
```

第三步：求私钥指数 `d`。

`d` 要满足：

```text
e * d ≡ 1 mod phi(n)
11d ≡ 1 mod 72
# 意思是 11*d 除以 72 余 1；d 是 11 在模 72 下的乘法逆元
```

用扩展欧几里得算法求逆元：

```text
72 = 11 * 6 + 6
# 先用 72 除以 11，余数是 6

11 = 6 * 1 + 5
# 再用 11 除以 6，余数是 5

6 = 5 * 1 + 1
# 再用 6 除以 5，余数是 1；出现 1 说明 11 和 72 互素，逆元存在
```

把 `1` 反向表示成 `72` 和 `11` 的线性组合：

```text
1 = 6 - 5
# 从最后一行 6 = 5*1 + 1 得到 1 = 6 - 5

1 = 6 - (11 - 6)
# 由 11 = 6*1 + 5 得到 5 = 11 - 6，把它代入

1 = 2*6 - 11
# 合并同类项，得到两个 6 减一个 11

1 = 2*(72 - 11*6) - 11
# 由 72 = 11*6 + 6 得到 6 = 72 - 11*6，再代入

1 = 2*72 - 13*11
# 展开后得到 1 = 2*72 - 13*11
```

从 `1 = 2*72 - 13*11` 可知：

```text
(-13) * 11 ≡ 1 mod 72
# 因为 2*72 在模 72 下等于 0，所以 -13 是 11 的逆元

d = -13 mod 72 = 59
# 私钥指数通常取 0 到 phi(n)-1 之间的正数；-13 加 72 得到 59
```

验证：

```text
11 * 59 = 649
649 = 72 * 9 + 1
# 649 除以 72 余 1，所以 11*59 ≡ 1 mod 72，d=59 正确
```

因此私钥为：

```text
(d, n) = (59, 91)
# d 是私钥指数，n 是同一个模数
```

第四步：加密明文 `M=9`。

```text
C = M^e mod n
C = 9^11 mod 91
# 用公钥指数 e=11 和模数 n=91 对明文数字 9 做模指数运算
```

用反复平方简化：

```text
9^2 = 81 mod 91
# 9 的平方是 81，小于 91，余数仍是 81

9^4 = 81^2 = 6561
6561 mod 91 = 9
# 91*72=6552，6561-6552=9，所以 9^4 mod 91 = 9

9^8 = (9^4)^2 = 9^2 = 81 mod 91
# 继续平方，得到 9^8 mod 91 = 81

9^11 = 9^8 * 9^2 * 9
# 11 的二进制分解可看作 8+2+1，所以拆成这三项相乘

9^11 mod 91 = 81 * 81 * 9 mod 91
81 * 81 = 6561 ≡ 9 mod 91
9 * 9 = 81
# 所以密文 C = 81
```

密文为：

```text
C = 81
```

第五步：解密验证。

```text
M = C^d mod n
M = 81^59 mod 91
# 用私钥指数 d=59 和模数 n=91 对密文 81 做模指数运算
```

计算时可以观察循环：

```text
81^2 mod 91 = 6561 mod 91 = 9
# 81 的平方模 91 得到 9

81^3 mod 91 = 81^2 * 81 mod 91 = 9 * 81 = 729 mod 91 = 1
# 91*8=728，729-728=1，所以 81^3 ≡ 1 mod 91

59 = 3*19 + 2
# 指数 59 可以拆成 19 个 3 再加 2

81^59 = (81^3)^19 * 81^2
# 因为 81^3 mod 91 = 1，前半部分等于 1

81^59 mod 91 = 1^19 * 9 = 9
# 解密后恢复明文 M=9
```

最终答案：

```text
n = 91
phi(n) = 72
d = 59
公钥 = (11, 91)
私钥 = (59, 91)
C = 81
解密后 M = 9
```

### 10.10 新 Word 例题 2：DES S 盒完整计算与逐步解释

题目条件：

```text
S 盒输入 = 011011
使用 S1 盒，查表得到第 1 行第 13 列的值为 5
```

DES S 盒规则：6 bit 输入，首尾两位定行，中间四位定列，查出的十进制值再转成 4 bit 二进制。

把输入位编号：

```text
输入 011011 = b1 b2 b3 b4 b5 b6
b1 = 0, b2 = 1, b3 = 1, b4 = 0, b5 = 1, b6 = 1
# 给每一位编号，是为了明确哪些位用于行号、哪些位用于列号
```

第一步：取第 1 位和第 6 位组成行号。

```text
行号 = b1b6 = 01_2 = 1
# b1 是 0，b6 是 1，组成二进制 01；二进制 01 等于十进制 1
```

注意：DES S 盒表的行通常按 `0,1,2,3` 编号。这里“行号为 1”表示二进制行值为 1，不要误写成第一行的行值 0。

第二步：取中间 4 位组成列号。

```text
列号 = b2b3b4b5 = 1101_2 = 13
# 中间四位是 1、1、0、1；二进制 1101 等于十进制 13
```

第三步：查 S1 盒。

```text
S1[1][13] = 5
# 行号为 1，列号为 13；按题目给出的 S1 表查到十进制值 5
```

第四步：把十进制输出转成 4 bit 二进制。

```text
5 = 0101_2
# S 盒输出固定为 4 bit；十进制 5 的二进制是 101，补足 4 位写成 0101
```

最终答案：

```text
S 盒输出 = 0101
```

易错点：

- 不要用前两位定行，行号一定是第 1 位和第 6 位。
- 不要把列号写成十进制 11；`1101_2` 等于 13。
- S 盒输出是 4 bit，不足 4 位要在前面补 0。
- 这道题是 DES 内部 S 盒计算，不是完整 DES 加密；完整 DES 明文输入和密文输出仍是 64 bit。

## 11. 临考易错点汇总

- 完整性不等于保密性；认证不等于不可抵赖。
- 对称密码速度快，但密钥分发困难。
- 公钥加密解决保密，不自动证明发送者身份。
- 私钥签名解决身份、完整性和不可抵赖，不提供保密。
- 数字证书绑定身份和公钥，不包含私钥。
- CRL/OCSP 检查的是证书是否已撤销，不只是是否过期。
- TLS 的服务器认证依赖证书链、域名、有效期、用途和撤销状态。
- SET 的双重数字签名是绑定订单摘要和支付摘要，不是把数据加密两次。
- DNSSEC 提供 DNS 记录认证和完整性，不加密 DNS 查询内容。
- SYN Flood 消耗半连接资源；Smurf 利用伪造源 IP 和广播放大。
- 防火墙只控制经过它的流量，不能解决所有内部威胁和应用漏洞。
- 区块链防篡改是“成本很高”，不是“绝对不可能”。
